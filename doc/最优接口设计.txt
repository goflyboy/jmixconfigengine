List<PartCategory> query(PartCategory category,PartConstraintReq constraintReq,FilterCondition attrWhereCondition ){ 
{
	 PartCategory resultPartCategory = category.clone();
	 List<PartCategory> resultSubPartCategory = new ArrayList();
	 if(!category.partCategoryMap.isEmpty()) {
		for(PartCategory pc: partCategoryMap.values()) {
			resultSubPartCategory.addAll(query(pc,constraintReq);
		}  
	 }	
	 resultPartCategory.addPartCategory(resultSubPartCategory);

	 //处理当前层part 
	 result = parseAttribute(constraintReq.attrCode,category.dynAttrSchemas), 解析出原始属性Capacity    
	 List<Part> filterParts = new ArrayList();
	 if(result.first.instType == 0) {//非实例属性（默认）
		filterParts = FilterExpressionExecutor.doSelect(category.partMap.values(),attrWhereCondition);  
	 }
	 else {//实例属性 
		for(Part part:	category.partMap.values()) {
			List<InstanceDynAttrValueItem> instValues = FilterExpressionExecutor.doSelect(part.getInstanceAttrs().getInstsValues(),attrWhereCondition);
			int sumAttrValue = 0;
			for(InstanceDynAttrValueItem instValue: instValues) {
					sumAttrValue += parseInt(instValue.getInstAttr(result.first.getCode())) //TODO类型处理
			}
		    Part cPart = part.clone();
	        cpart.addAttr(result.first.getCode(),String.valueOf(sumAttrValue));
			filterParts.add(cPart);
		} 
	 }
	 resultPartCategory.addPart(filterParts); 
}

4、固态硬盘优先匹配高速率容量，用机械硬盘增配低速率容量
所以，“固态硬盘优先匹配高速率容量”可能意味着：在满足5400速率需求时，优先使用转速更高的实例分组（即7200/5400，因为它还支持7200，所以是更高速的）来提供容量，当这种高速实例分组不够时，再使用只支持5400的实例分组。

建模步骤：

将固态硬盘的实例分组分为两类：高速组（转速包含7200或9000，且同时包含5400，即7200/5400）和低速组（只包含5400）。注意，根据数据，固态硬盘中没有只包含7200的，所以高速组就是7200/5400，低速组是5400。

优先使用高速组的固态硬盘，当高速组的固态硬盘用尽（达到最大数量）仍不满足容量需求时，再使用低速组的固态硬盘。

当固态硬盘的所有实例分组（高速和低速）都用尽后，仍不满足容量需求，再使用机械硬盘（机械硬盘都是5400）。

因此，我们需要按照以下顺序使用硬盘：
第一优先级：固态硬盘高速组（转速7200/5400）
第二优先级：固态硬盘低速组（转速5400）
第三优先级：机械硬盘（转速5400）

但是，在约束建模中，我们通常通过条件约束来实现。不过，这里我们可以用另一种方式：通过权重和约束结合。

然而，如果我们已经用权重来体现优先级，那么求解器会自动优先选择固态硬盘，直到固态硬盘不够用时才会选择机械硬盘。
但是，这并不能严格保证“固态硬盘满配后才用机械硬盘”，因为如果固态硬盘数量没有达到上限，但是已经满足容量需求，那么就不会用机械硬盘，这是合理的
