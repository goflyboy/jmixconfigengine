# RFC-0005-0.1: 南向接口解耦补丁

> 状态: 草案(Draft)
> 日期: 2026-05-17
> 修补文档: `doc/RFC-0005-North-South-Interface-Decoupling.md`
> 相关文档: `doc/RFC-0006-North-Interface-Refactor.md`, `doc/CORE-DESIGN.md`, `doc/ACCEPTANCE.md`

---

## 设计决议摘要

本 RFC 是 `RFC-0005` 的补丁版本。`RFC-0005` 已经把南向接口从测试基类和部分内部实现中抽离出来，但仍未完全达到“产品算法只依赖稳定领域接口”的目标。本补丁将南向接口进一步收敛为以 `ModuleCPModel` 和 `ModuleInstView` 为核心的领域模型，并把 Google OR-Tools 类型完全隔离到实现层。

| 主题 | 决议 |
| --- | --- |
| CP 接口拆分 | `AlgCPConstraint`、`AlgCPLinearExpr`、`AlgCPModel`、`PartAlgCPLinearExpr` 改为纯接口，实现类统一命名为 `XxxImpl` |
| Google 类型解耦 | 南向 API 和产品算法不再暴露 `IntVar`、`BoolVar`、`Literal`、`Constraint` 等 OR-Tools 类型，改用 `AlgCPIntVar`、`AlgCPBoolVar`、`AlgCPLiteral` 等 facade |
| 约束模型命名 | `ConstraintModel` 更名并收敛为 `ModuleCPModel`，体现“模块约束编程模型”的领域概念 |
| PartCategory 能力 | `IPartCategoryFunction`、`PartCategoryFunction` 收敛为 `PartCategoryCPModel` / `PartCategoryCPModelImpl` |
| 变量注册 | 删除 `ConstraintVarRegistry` 概念，将 `para`、`part`、`partCategory` 等查找方法直接放入 `ModuleCPModel` |
| 算法基类 | `ConstraintAlgBase` 更名为 `ModuleAlgBase`，产品算法通过 `model()` 使用全部南向能力 |
| 算法版本注解 | `AlgorithmApiVersion` 不属于南向接口，移动到 tools 注解包，仅作为源码生成和 artifact 构建辅助 |
| API 兼容策略 | 当前补丁仍作为 1.0 试验期处理；1.0 正式发布后，南向接口只能新增，不能改名、改类型或删除 |
| 方法级版本标记 | 南向接口方法增加 `@SouthApiSince` 标记，用于识别每个 API 方法从哪个版本开始可用 |
| 实现基类 | `ModuleBaseAlgImpl` 调整为实现 `ModuleCPModel` 和 `ModuleInstView`，作为引擎内部实现承载者 |
| POST 实例访问 | `ModuleInstAccessor` 被 `ModuleInstView` 取代，删除 accessor 形态 |
| 冗余类型 | 删除或迁移 `PartCategoryAlg`、`ModuleAlg`、`IModuleAlg`、`IConstraintFunction`、`ConstraintFunction`、`ConstraintContext`、`ConstraintAlgorithm` |
| 测试迁移 | 所有测试用例按 `model().xxx` 和 `AlgCPXxx` facade 方式整改 |

---

## 1. 摘要

当前南向接口仍存在两类问题：

1. 南向接口和 Google OR-Tools 解耦不干净。算法代码仍可能直接使用 `IntVar`、`BoolVar`、`Literal`，`AlgCPModel` 等类型本身也是实现类而非稳定接口。
2. 约束接口的领域概念不够清晰。`ConstraintModel`、`ConstraintFunction`、`PartCategoryFunction`、`ConstraintVarRegistry` 等类型数量较多，但没有形成以 Module 为根的统一约束编程模型，导致使用者需要理解过多内部层次。

本补丁将南向 API 的核心入口定义为：

```java
public interface ModuleCPModel extends AlgCPModel, PartCategoryCPModel {
    ParaVar para(String code);
    PartVar part(String code);
    PartCategoryVar partCategory(String code);
}
```

产品算法统一通过 `model()` 使用约束能力、PartCategory 聚合能力和变量查找能力；POST 阶段统一通过 `ModuleInstView` 访问单个配置解。

---

## 2. 动机

### 2.1 耦合度问题

`RFC-0005` 已经提出要把产品算法从 `ConstraintAlgImplTestBase`、`ModuleAlgImpl` 和内部变量对象中解耦出来，但当前实现仍有残留耦合：

- `AlgCPModel`、`AlgCPLinearExpr`、`AlgCPConstraint`、`PartAlgCPLinearExpr` 是实现类，并直接依赖 OR-Tools。
- 南向算法样例和部分测试仍导入 `com.google.ortools.sat.BoolVar`、`IntVar`、`Literal`。
- `southinf.var.PartCategoryVar` 等 facade 仍可能直接引用 `impl.algmodel` 实现类型。
- `ConstraintAlgBase` 仍继承 `ModuleAlgImpl`，产品算法入口和内部执行实现没有彻底分层。
- POST 访问仍通过 `ModuleInstAccessor` 这种过程式访问器间接操作结果实例。

这些耦合会让产品算法跟随求解器封装和内部执行结构变动而变动，无法形成稳定南向契约。

### 2.2 接口设计与领域模型问题

`RFC-0005` 引入了 `ConstraintModel`、`ConstraintContext`、`ConstraintVarRegistry` 等类型，但从产品算法视角看，约束编程模型的真实领域根应该是 Module：

- Module 级规则需要访问模块参数、模块部件、所有 PartCategory 聚合能力。
- PartCategory 级规则需要在当前分类上下文中访问筛选、求和、选中数量等能力。
- POST 规则需要访问当前 `ModuleInstView`，而不是知道底层 `ModuleInst` 或 `ModelArgInPlay`。

因此，本补丁将接口收敛到 `ModuleCPModel`、`PartCategoryCPModel`、`ModuleInstView` 三个核心领域入口。

---

## 3. 范围

本 RFC 覆盖：

- 南向 CP facade 接口和实现类拆分。
- OR-Tools 类型从南向接口中移除。
- `ConstraintModel` 到 `ModuleCPModel` 的领域化改造。
- `PartCategoryFunction` 到 `PartCategoryCPModel` 的改造。
- `ConstraintAlgBase` 到 `ModuleAlgBase` 的改造。
- `ModuleInstAccessor` 到 `ModuleInstView` 的替换。
- 冗余南向类型删除。
- 测试用例和代码规则样例迁移。

本 RFC 不覆盖：

- 北向 `calculate` / `infer` / `postCalculate` 接口设计，见 `RFC-0006`。
- 替换 Google OR-Tools 求解器。
- 改写业务领域模型 `bmodel` 和结果模型 `cmodel` 的数据结构。

## 5. 设计方案

### 5.1 CP facade 拆分

现有实现类拆分如下：

| 当前类型 | 目标接口 | 目标实现 |
| --- | --- | --- |
| `AlgCPConstraint` | `AlgCPConstraint` | `AlgCPConstraintImpl` |
| `AlgCPLinearExpr` | `AlgCPLinearExpr` | `AlgCPLinearExprImpl` |
| `AlgCPModel` | `AlgCPModel` | `AlgCPModelImpl` |
| `PartAlgCPLinearExpr` | `PartAlgCPLinearExpr` | `PartAlgCPLinearExprImpl` |

接口示意：

```java
public interface AlgCPLiteral {
    String name();
    AlgCPLiteral not();
}

public interface AlgCPLinearArgument {
    String name();
}

public interface AlgCPIntVar extends AlgCPLinearArgument {
}

public interface AlgCPBoolVar extends AlgCPIntVar, AlgCPLiteral {
}

public interface AlgCPConstraint {
    AlgCPConstraint onlyEnforceIf(AlgCPLiteral condition);
    AlgCPConstraint onlyEnforceIf(AlgCPLiteral... conditions);
    int index();
}

public interface AlgCPLinearExpr extends AlgCPLinearArgument {
    AlgCPLinearExpr name(String name);
    AlgCPLinearExpr addTerm(AlgCPIntVar var, long coefficient);
    AlgCPLinearExpr addTerm(AlgCPBoolVar var, long coefficient);
    AlgCPLinearExpr addConstant(long value);
    boolean isEmpty();
}

public interface PartAlgCPLinearExpr extends AlgCPLinearExpr {
    @Override
    PartAlgCPLinearExpr name(String name);
}
```

`AlgCPModel` 只暴露 facade 类型：

```java
public interface AlgCPModel {
    AlgCPIntVar newIntVar(long left, long right, String name);
    AlgCPIntVar newIntVarFromDomain(long[] values, String name);
    AlgCPBoolVar newBoolVar(String name);

    AlgCPLinearExpr newLinearExpr(String name);
    PartAlgCPLinearExpr newPartLinearExpr(String name);

    AlgCPConstraint addBoolAnd(AlgCPLiteral[] literals);
    AlgCPConstraint addBoolOr(AlgCPLiteral[] literals);
    AlgCPConstraint addExactlyOne(AlgCPLiteral[] literals);
    AlgCPConstraint addAtMostOne(AlgCPLiteral[] literals);
    AlgCPConstraint addImplication(AlgCPLiteral left, AlgCPLiteral right);

    AlgCPConstraint addEquality(AlgCPLinearArgument left, long right);
    AlgCPConstraint addEquality(AlgCPLinearArgument left, AlgCPLinearArgument right);
    AlgCPConstraint addEquality(AlgCPLinearExpr left, long right);

    AlgCPConstraint addLessOrEqual(AlgCPLinearArgument left, long right);
    AlgCPConstraint addLessOrEqual(AlgCPLinearExpr left, long right);
    AlgCPConstraint addGreaterOrEqual(AlgCPLinearArgument left, long right);
    AlgCPConstraint addGreaterOrEqual(AlgCPLinearExpr left, long right);

    void minimize(AlgCPLinearExpr expr);
    void maximize(AlgCPLinearExpr expr);
}
```

实现类负责在内部 unwrap 到 OR-Tools 类型：

```java
final class AlgCPBoolVarImpl implements AlgCPBoolVar {
    private final com.google.ortools.sat.BoolVar delegate;
}
```

### 5.2 ModuleCPModel

`ConstraintModel` 更名为 `ModuleCPModel`，并吸收变量查找能力：

```java
public interface ModuleCPModel extends AlgCPModel, PartCategoryCPModel {
    ParaVar para(String code);
    PartVar part(String code);
    PartCategoryVar partCategory(String code);
}
```

`ConstraintVarRegistry` 删除，不再作为独立接口暴露。原 `vars().para(...)` 迁移为 `model().para(...)`。

迁移示例：

```java
// before
ParaVar color = vars().para("Color");

// after
ParaVar color = model().para("Color");
```

### 5.3 PartCategoryCPModel

原 `IPartCategoryFunction` / `PartCategoryFunction` 改为领域化的 `PartCategoryCPModel`：

```java
public interface PartCategoryCPModel {
    PartAlgCPLinearExpr sum4Quantity(String attrCode, String filterCondition);
    AlgCPLinearExpr sum4Selected(String filterCondition);
    List<PartVar> partVars();
    List<PartVar> partVars(String filterCondition);
}
```

作用域规则：

- 在 Module 级规则中，`model().sum4Quantity(...)` 按当前规则声明或显式参数选择 Module 下的 PartCategory 范围。
- 在 PartCategory 级规则中，`model().sum4Selected(...)` 默认作用于当前 `fatherCode` 对应的 PartCategory。
- 不再允许直接调用继承来的 `sum4Quantity(...)`、`sum4Selected(...)`，必须经由 `model()`。

### 5.4 算法基类

`ConstraintAlgBase` 更名为 `ModuleAlgBase`。产品算法继承 `ModuleAlgBase`，不继承 `ModuleBaseAlgImpl` 或 `ModuleAlgImpl`。

```java
public abstract class ModuleAlgBase implements ModuleInstView {
    private ModuleCPModel constraintModel;
    private ModuleInstView currentInst;

    protected final ModuleCPModel model() {
        return constraintModel;
    }

    protected final ModuleInstView currentInst() {
        return currentInst;
    }
}
```

`ModuleBaseAlgImpl` 是引擎内部实现，调整为：

```java
class ModuleBaseAlgImpl implements ModuleCPModel, ModuleInstView {
    private ModuleCPModel constraintModel;
    private ModuleInstView currentInst;
}
```

约束：

- 产品算法不直接依赖 `ModuleBaseAlgImpl`。
- 产品算法只感知 `ModuleAlgBase`、`ModuleCPModel`、`ModuleInstView` 和变量 facade。
- `ModuleAlgBase.model()` 是南向约束能力唯一入口。

### 5.5 AlgorithmApiVersion 归属

`AlgorithmApiVersion` 从 `com.jmix.executor.southinf` 移到 tools 注解包：

```java
package com.jmix.tool.bbuilder.anno;

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.TYPE)
public @interface AlgorithmApiVersion {
    String southApiVersion();
    String algorithmVersion() default "";
}
```

定位：

- 它是源码生成、模型构建和 artifact 构建阶段的辅助注解。
- 它不是产品算法运行期必须依赖的南向接口。
- 运行期版本判定仍以 `ModuleAlgArtifact.southApiVersion` 为准。
- 若源码注解和 artifact 版本同时存在，artifact 版本优先；源码注解只用于生成期补全或校验。

### 5.6 南向 API 版本与兼容性

当前 `RFC-0005-0.1` 仍处于南向接口 1.0 试验阶段：

- 当前补丁不升级 `southApiVersion`，初始版本仍为 `1.0`。
- 试验阶段不要求兼容旧的 1.0 测试用例，测试和样例可以一次性迁移到新南向接口。
- 一旦 1.0 正式发布，从 1.0 之后开始建立严格兼容约束。

1.0 正式发布后的硬性约束：

- 已发布接口方法不能改名。
- 已发布接口方法的参数类型、返回类型、异常契约不能修改。
- 已发布接口方法不能删除。
- 已发布接口类型不能移动包名。
- 新能力只能通过新增接口类型、新增方法或新增默认方法提供。

版本演进示例：

```text
1.1:
  ModuleCPModel#newBoolVar(String)
  ModuleCPModel#addBoolAnd(AlgCPLiteral[])

1.2:
  ModuleCPModel#addAllDifferent(AlgCPLinearArgument[])
  PartCategoryCPModel#sum4SelectedQuantity(String)
```

当 `southApiVersion` 从 `1.1` 增加到 `1.2` 时，工具需要能回答：

- 哪些 API 是 1.1 已有的。
- 哪些 API 是 1.2 新增的。
- 是否有 1.1 API 被改名、改类型或删除。

### 5.7 方法级 API 版本标记

新增南向接口方法级版本注解：

```java
package com.jmix.executor.southinf.version;

@Retention(RetentionPolicy.RUNTIME)
@Target({ElementType.TYPE, ElementType.METHOD})
public @interface SouthApiSince {
    String value();
}
```

可选提供版本常量，避免字符串散落：

```java
package com.jmix.executor.southinf.version;

public final class SouthApiVersion {
    public static final String V1_0 = "1.0";
    public static final String V1_1 = "1.1";
    public static final String V1_2 = "1.2";

    private SouthApiVersion() {
    }
}
```

接口标记示例：

```java
@SouthApiSince(SouthApiVersion.V1_0)
public interface ModuleCPModel extends AlgCPModel, PartCategoryCPModel {

    @SouthApiSince(SouthApiVersion.V1_0)
    AlgCPBoolVar newBoolVar(String name);

    @SouthApiSince(SouthApiVersion.V1_0)
    AlgCPConstraint addBoolAnd(AlgCPLiteral[] literals);

    @SouthApiSince(SouthApiVersion.V1_2)
    AlgCPConstraint addAllDifferent(AlgCPLinearArgument[] expressions);
}
```

标记规则：

- 每个南向 API 接口和方法都必须显式标记 `@SouthApiSince`。
- 1.0 试验期形成的最终基线统一标记为 `@SouthApiSince("1.0")`。
- 后续新增方法按新增版本标记，例如 `@SouthApiSince("1.2")`。
- 禁止通过修改 `@SouthApiSince` 掩盖 API 变更；老方法的版本标记一旦发布不可修改。
- 实现类不要求重复标记，兼容性检查以接口为准。

自动化检查工具应扫描 `southinf` 接口并生成 API manifest：

```json
{
  "southApiVersion": "1.2",
  "methods": [
    {
      "owner": "com.jmix.executor.southinf.ModuleCPModel",
      "name": "newBoolVar",
      "descriptor": "(java.lang.String)com.jmix.executor.southinf.cp.AlgCPBoolVar",
      "since": "1.0"
    },
    {
      "owner": "com.jmix.executor.southinf.ModuleCPModel",
      "name": "addAllDifferent",
      "descriptor": "(com.jmix.executor.southinf.cp.AlgCPLinearArgument[])com.jmix.executor.southinf.cp.AlgCPConstraint",
      "since": "1.2"
    }
  ]
}
```

manifest 存放路径：

```text
doc/api/south-api-<version>.json
```

示例：

```text
doc/api/south-api-1.0.json
doc/api/south-api-1.1.json
doc/api/south-api-1.2.json
```

manifest 对比规则：

- 当前版本必须包含上一版本 manifest 中的全部接口方法。
- 上一版本方法的 owner、name、descriptor、since 必须完全一致。
- 当前版本新增方法的 `since` 必须等于当前 `southApiVersion`。
- 禁止出现没有 `@SouthApiSince` 的南向接口方法。

### 5.8 ModuleInstView 替代 ModuleInstAccessor

删除 `ModuleInstAccessor` 和 `ModuleInstAccessorImpl`。POST 规则通过 `ModuleInstView` 访问和写回当前配置解。

```java
@CodeRuleAnno(calcStage = CalcStage.POST)
private void postRule() {
    PartCategoryInstSumView drive = partCategorySum("drive");
    int sum = drive.sumDynAttr4Int("Capacity");

    parameter("pDriveSumCapacity").setValue(String.valueOf(sum));
}
```

实现层基于最新 `ModuleInstView` 接口重新实现 `ModuleInstViewImpl`，不再保留 accessor 概念。

### 5.9 删除冗余南向类型

以下类型不再作为南向 API 保留：

```text
PartCategoryAlg
ModuleAlg
IModuleAlg
IConstraintFunction
ConstraintFunction
ConstraintContext
ConstraintAlgorithm
ConstraintVarRegistry
ConstraintModel
AlgorithmApiVersion
```

迁移关系：

| 旧类型 | 新类型 |
| --- | --- |
| `ConstraintModel` | `ModuleCPModel` |
| `ConstraintVarRegistry` | `ModuleCPModel` 内置查找方法 |
| `IPartCategoryFunction` | `PartCategoryCPModel` |
| `PartCategoryFunction` | `PartCategoryCPModelImpl` |
| `ConstraintAlgBase` | `ModuleAlgBase` |
| `ConstraintContext` | `ModuleAlgBase` 绑定的 `ModuleCPModel` / `ModuleInstView` |
| `ModuleInstAccessor` | `ModuleInstView` |
| `ModuleInstAccessorImpl` | 重新实现的 `ModuleInstViewImpl` |
| `com.jmix.executor.southinf.AlgorithmApiVersion` | `com.jmix.tool.bbuilder.anno.AlgorithmApiVersion` |

### 5.10 使用样例

#### 样例 1: Module 级 PartCategoryCPModel 函数

```java
// before
PartAlgCPLinearExpr totalCapacity = model.sum4Quantity("Capacity", "").name("totalCapacity");
PartAlgCPLinearExpr objectiveExpr = model.newPartLinearExpr("ObjectiveFun");

// after
PartAlgCPLinearExpr totalCapacity = model().sum4Quantity("Capacity", "").name("totalCapacity");
PartAlgCPLinearExpr objectiveExpr = model().newPartLinearExpr("ObjectiveFun");
```

#### 样例 2: PartCategory 级 PartCategoryCPModel 函数

```java
// before
@CodeRuleAnno(fatherCode = "drive", normalNaturalCode = "固态硬盘必须配置同一种，并且最多配置2块")
private void logicB1() {
    AlgCPLinearExpr sdTypeSumNum = sum4Selected("Type=sd").name("sdTypeSumNum");
    model.addLessOrEqual(sdTypeSumNum, 1);
}

// after
@CodeRuleAnno(fatherCode = "drive", normalNaturalCode = "固态硬盘必须配置同一种，并且最多配置2块")
private void logicB1() {
    AlgCPLinearExpr sdTypeSumNum = model().sum4Selected("Type=sd").name("sdTypeSumNum");
    model().addLessOrEqual(sdTypeSumNum, 1);
}
```

#### 样例 3: BoolVar 与 AlgCPModel 解耦

```java
// before
@CodeRuleAnno
public void addConstraintRule2() {
    BoolVar redAndSmall = newBoolVar("rule2_redAndSmall");

    model.addBoolAnd(new Literal[] {
            this.colorVar.option("Red").selectedVar(),
            this.sizeVar.option("Small").selectedVar()
    }).onlyEnforceIf(redAndSmall);
}

// after
@CodeRuleAnno
public void addConstraintRule2() {
    AlgCPBoolVar redAndSmall = model().newBoolVar("rule2_redAndSmall");

    model().addBoolAnd(new AlgCPLiteral[] {
            this.colorVar.option("Red").selectedVar(),
            this.sizeVar.option("Small").selectedVar()
    }).onlyEnforceIf(redAndSmall);
}
```

注意：最终目标明确为 `AlgCPBoolVar` 和 `AlgCPLiteral`，不再允许产品算法使用 OR-Tools 的 `BoolVar` 和 `Literal`。

## 6. 测试迁移规范

所有南向算法测试和代码规则测试按以下规则整改：

| 旧写法 | 新写法 |
| --- | --- |
| `model.addLessOrEqual(...)` | `model().addLessOrEqual(...)` |
| `model.newBoolVar(...)` | `model().newBoolVar(...)` |
| `newBoolVar(...)` | `model().newBoolVar(...)` |
| `newIntVar(...)` | `model().newIntVar(...)` |
| `sum4Quantity(...)` | `model().sum4Quantity(...)` |
| `sum4Selected(...)` | `model().sum4Selected(...)` |
| `newPartLinearExpr(...)` | `model().newPartLinearExpr(...)` |
| `BoolVar` | `AlgCPBoolVar` |
| `IntVar` | `AlgCPIntVar` |
| `Literal` | `AlgCPLiteral` |
| `ModuleInstAccessor` | `ModuleInstView` |

测试整改范围至少包括：

## 9. 风险与兼容策略

### 9.1 源码兼容风险

本补丁会破坏仍使用 `BoolVar`、`IntVar`、`Literal`、`model.xxx` 字段、`newBoolVar(...)` 继承方法的算法源码。

缓解：

- 统一迁移测试用例和样例，确保生成器按新接口生成。
- 当前阶段仍属于 1.0 试验期，不保留源码兼容层。
- 1.0 正式发布后开始执行“只能新增，不能修改或删除”的兼容规则。

### 9.2 实现复杂度风险

`AlgCPBoolVar`、`AlgCPLiteral`、`AlgCPLinearArgument` 和 OR-Tools 类型之间需要 unwrap，适配层可能膨胀。

缓解：

- unwrap 逻辑只允许存在于 `impl.algmodel`。
- 每个 facade impl 提供包内可见的 delegate 访问方法。
- 实现单测覆盖 `not()`、`onlyEnforceIf()`、`addBoolAnd()`、线性表达式构建等核心路径。

### 9.3 命名迁移风险

当前代码已经存在 `ModuleBaseAlgImpl`、`ModuleAlgImpl`、`PartCategoryAlgImpl` 等内部类型，和新增领域接口可能发生命名混乱。

缓解：

- 产品算法侧只暴露 `ModuleAlgBase`。
- 引擎内部实现类保留 `Impl` 后缀。
- 对外文档中不再出现 `ConstraintModel`、`ConstraintContext`、`IModuleAlg`。

### 9.4 API 版本标记遗漏风险

后续新增方法如果没有 `@SouthApiSince`，工具就无法判断它属于哪个版本。

缓解：

- API 兼容性测试默认扫描全部南向接口方法。
- 缺少 `@SouthApiSince` 的方法直接导致测试失败。
- manifest 作为版本发布产物随代码一起提交。

## 10. 已确认决策

1. 当前补丁仍属于 1.0 试验阶段，不升级 `southApiVersion`，初始版本仍为 `1.0`。

2. 试验阶段不考虑旧版 1.0 测试用例兼容；所有测试和样例一次性迁移到 `model()` 和 `AlgCPXxx` facade。

3. 1.0 正式发布后开始考虑兼容性；后续版本只能新增 API，不能改名、改类型或删除已发布 API。

4. 南向接口方法需要增加版本标记，使用 `@SouthApiSince` 标明方法从哪个版本开始支持。

5. 工具类或自动化测试需要能生成 API manifest，并快速比较当前版本相对上一版本新增了哪些接口。

6. 样例 3 的最终目标类型是 `AlgCPBoolVar`，不是 OR-Tools `BoolVar`。

7. `ModuleInstAccessor` 和 `ModuleInstAccessorImpl` 都删除；`ModuleInstViewImpl` 基于最新接口重新实现。

8. `ModuleAlgBase` 是算法南向接口唯一继承类，且必须是非常薄的一层；不能在其中增加业务逻辑，业务逻辑由实现类承担。

9. `ModuleBaseAlgImpl` 是引擎内部实现类，产品算法不允许继承。

10. 实现层单测允许导入 OR-Tools，但必须限制在 `impl` 或专门 adapter 测试中，不能作为产品算法样例。

11. 方法级版本标记命名确定为 `@SouthApiSince`。

12. API manifest 文件固定存放在 `doc/api/south-api-<version>.json`。
