# AI 辅助开发方法论对比分析

**文档状态**：第四版（新增哲学基础章节、补全本体论认知局限分析）
**日期**：2026-05-03
**作者**：Claude (AI Coding Assistant)
**目的**：对比 RFC 驱动开发（自研方法）与业界主流 AI 开发方法论（SDD/VSDD/Superpowers），分析异同、优缺点及可借鉴点，并为此方法论命名

## 1. 概述：被比较的方法论

### 1.1 RFC 驱动开发（RFD）

本文档分析的两个自研 skills，统称为 RFC 驱动开发方法：

- `interactive-rfc-generation`（交互式 RFC 生成）：结对编程式 RFC 文档生成工作流，4 步迭代：AI 写初稿 → AI 提出疑问 → 用户解答 → 修改完善 → AI 评审。
- `rfc-driven-development`（RFC 驱动开发）：端到端开发流程，读取 RFC 文档 → 代码开发 → 测试用例编写 → 测试验证 → 自动创建分支提交。

#### 1.1.1 方法论核心哲学

RFD 强调人类思考的**非线性和闪现性**——人的思维是碎片式的、有限次的，不像文档那样可以线性输出。AI 的核心作用不是被动响应一段 prompt，而是在人脑闪现出想法的当下，主动提问、追问、引导，帮助用户把那些稍纵即逝的想法**结构化**成一份 RFC 文档。整个方法论的目标是：**生产 RFC**。

#### 1.1.2 RFC 生成的三大输入依赖

RFC 并非凭空产生，其生成过程严格依赖三个输入条件：

**依赖一：项目核心架构与验收标准（脚手架）**

AI 在生成 RFC 初稿前，必须读取项目的基础规范文件：
- `doc/CORE-DESIGN.md`：系统的核心设计文档，定义架构分层、关键接口、扩展点和技术选型边界
- `doc/ACCEPTANCE.md`：项目的验收准则，包含测试用例设计原则、开发流程原则和代码风格要求

这些文件是 RFC 的**技术约束空间**——RFC 中的设计方案不能违背核心架构，验收准则决定了 RFC 必须包含的测试用例格式。

**依赖二：用户的原始零星想法（原材料）**

用户提供的想法可能是：
- 一个产品意图的描述（"我想让系统支持后置计算"）
- 一个具体的问题场景（"当 drive 分类有多个实例时，POST 规则应该怎么访问"）
- 一个模糊的方向（"我想把 PRE/MID/POST 三阶段分开"）

这些想法不需要完整、不需要线性、不需要技术化。AI 的职责就是通过追问，把这些原材料塑造成结构化的 RFC。

**依赖三：迭代式的初稿与反馈循环（催化剂）**

AI 生成初稿后，不是把文档丢给用户等待结论，而是**主动提出设计决策点**（Q1、Q2、Q3...），等待用户明确答复后再继续完善。这种迭代过程本身就是人类参与 RFC 创作的核心环节——人在这个过程中承担了两重角色：

1. **需求规格的理解者**：用户最清楚业务意图，AI 无法代替判断"这个方向对不对"
2. **架构实现的审核者**：RFC 必须在项目架构约束内可行，这要求审核者同时具备架构知识

因此，对 RFC 审核者的要求是比较高的——他既要是业务专家，又要对项目的核心设计有足够的理解。

#### 1.1.3 软件本体的多视图愿景

RFD 的长期目标是：将软件视为一个**大型本体对象**，这个本体有多个视图（View），每个视图都是对本体的不同投影：

| 视图 | 描述 | 成熟度 |
|------|------|--------|
| **代码视图** | 运行时代码包，JAR/WAR 等可执行产物 | 当前 |
| **功能特性视图** | 按功能模块组织的特性树 | 规划中 |
| **架构设计视图** | CORE-DESIGN.md 定义的分层架构 | 当前 |
| **需求维护视图** | RFC 文档集合，按需求维度组织 | 当前 |

这些视图在语义上是**等价的**——它们都是同一个本体的不同投影。对需求维护者来说，可以从需求视图出发，沿架构设计视图深入，落到详细设计，最终生成代码视图；反之，代码的任何改动，也应该能在其他视图中追溯其语义影响。

**开放问题**：运行时代码包的本体定位。当代码编译打包后，JAR/WAR 本身是什么？是本体的投影终点，还是另一个独立的本体节点？这涉及到"编译后的产物"与"源代码"之间的语义等价性建模，目前尚未解决。

### 1.2 Spec-Driven Development（SDD）

Spec-Driven Development 是 2026 年在 AI 编程领域兴起的结构化方法论。其核心哲学是**规范（Spec）是真相来源，代码是一次性产物**。流程分为四阶段：

```
Specify（规范定义）→ Plan（技术规划）→ Break Down（任务分解）→ Implement（实现）
```

SDD 强调**数据构建方式**作为其本质——Spec 是驱动代码生成的原材料，代码是从规范衍生的产物。

### 1.3 Verified Spec-Driven Development（VSDD）

Verified Spec-Driven Development 是 SDD 的进阶版本，融合了 TDD 和 Verification-Driven Development，形成六阶段流水线：

```
Spec Crystallization（规范精炼）→ Test-First Implementation（测试优先实现）
→ Adversarial Review（对抗性评审）→ Feedback Integration（反馈整合）
→ Formal Hardening（正式加固）→ Convergence（收敛确认）
```

其核心创新是引入了**对抗性评审（Adversarial Review）**——由不同 AI 模型实例扮演"批评者"角色，专门挑刺。

### 1.4 Superpowers

Superpowers 是一个开源插件系统（GitHub 139k+ stars），为 Claude Code 等 AI 编程工具强制执行七阶段结构化方法论：

```
Brainstorming（头脑风暴）→ Isolation（隔离工作）→ Planning（规划）
→ Execution（执行）→ Testing（测试）→ Review（评审）→ Completion（完成）
```

## 3. 本质差异分析

### 3.1 核心哲学的根本分歧

**RFD 的哲学**：`人的思维是闪现的、有限的、非线性的。AI 的任务是捕捉并结构化这些闪现，将其固化为 RFC。代码与 RFC 同等权威，开发过程中通常不修改 RFC。软件是多视角的本体，各视图语义等价`。

这是所有路线中**唯一从人类认知科学出发的**方法论——它承认人类思维的碎片性，并以此为方法论设计的起点。

**SDD 的哲学**：`规范是真相，代码是产物（可丢弃）`。强调数据的构建方式——Spec 作为原材料驱动代码生成。

**VSDD 的哲学**：`规范定义"做什么"，测试强制"怎么做"，对抗评审确保"无遗漏"`。

**Superpowers 的哲学**：`人机协作的纪律约束`。强制 AI 在每一步停下来等待人类确认，人类是最终的决策者和审批者。

### 3.2 人机交互模式的差异

| 维度 | RFD | SDD | VSDD | Superpowers |
|------|-----|-----|------|-------------|
| 交互时机 | RFC 生成阶段（深度） | Plan 阶段（一次性） | 多阶段多次 | 每个任务前 |
| 交互方式 | 对话式问答（AI 主动追问） | 文档式 | 评审报告 | 任务审批 |
| 用户角色 | 闪现想法的持有者 + 架构审核者 | 需求提供者 | 架构师（仅战略决策） | 审批者（全程把关） |
| AI 主动性 | **极高（主动提问引导用户）** | 低（被动响应规范） | 中（生成 + 等待） | 低（每步等待确认） |

### 3.3 规范文档的角色定位

| 方法论 | 规范的角色 | 规范是否可执行 | 规范与代码的权威关系 |
|--------|-----------|--------------|-------------------|
| RFD | 人类闪现想法的结构化凝固；软件本体的需求视图 | 间接（通过测试用例） | **代码、语义、RFC 三者同等权威** |
| SDD | 唯一真相来源 + 代码生成原材料 | 否 | 规范 > 代码 |
| VSDD | 行为契约 | 间接（通过 TDD 测试） | 三层同等重要 |
| Superpowers | 规划工具 | 否 | 人 > 规范 > AI 实现 |

**RFD 的独特之处**：代码、语义、RFC 文档三者处于同等地位。这意味着：
- 开发过程中发现 RFC 与代码不一致，**不是改 RFC**，而是确保代码正确理解 RFC
- RFC 是人类想法的凝固表达，不随实现过程而贬值
- 代码是对 RFC 的忠实翻译，而非可丢弃的临时产物

### 3.4 质量保障机制的差异

| 方法论 | 质量保障机制 | TDD 的角色 | 关键缺陷 |
|--------|------------|-----------|---------|
| RFD | 注解驱动测试（强制要求写入 RFC 验收准则） | TDD 是质量保障的核心手段，**在 RFC 的约束下增强执行** | 无强制门控；评审是同一模型 |
| SDD | 无正式质量门控 | 提及但未强制 | 无强制测试、无评审 |
| VSDD | 六阶段流水线强制门控 | TDD 强制，测试优先于实现 | 流程过重；规范错误时 TDD 方向也会错 |
| Superpowers | 两阶段评审 + Completion 门控 | TDD 强制 | 同一 AI 评审；交互过频 |

### 3.5 人类认知模型的差异

这是 RFD 与所有其他方法论的根本分歧所在：

| 方法论 | 人类认知假设 | 规范生成方式 |
|--------|------------|------------|
| **RFD** | **人类思维是碎片式闪现的、有限的、非线性的** | AI 通过追问将闪现的想法结构化 |
| SDD | 人类能够线性地输出完整需求 | 用户一次性写出规范，AI 补充细节 |
| VSDD | 人类能够提供行为契约 | 用户写 5-10 句行为规范，AI 精炼 |
| Superpowers | 人类能够分解任务并逐项审批 | AI 生成任务列表，人类审批 |

### 3.6 现有本体方法论的认知局限

RFD 提出的"软件本体多视图"愿景，并非对传统本体论方法的简单继承。恰恰相反，它的提出正是因为发现了传统本体方法论在以下两个关键维度上的根本缺失。

#### 3.6.1 维度一：视角依赖性（Perspectival Nature）

**传统本体论的假设**：存在一个客观的、不依赖于观察者视角的事物本质。本体的核心关系是 `IS-A`（"书包是一个容器"），同一个实体在不同观察者眼中应该被映射到同一个本体节点。

**现实中的认知现象**：同一个实体，在不同观察者的视角下，被理解为完全不同的事物：

- 在学生眼中：书包是一个**收纳工具**（装书本、装文具）
- 在工厂工人眼中：书包是一条**生产线上的产品**（物料、工艺、工序）
- 在物流工人眼中：书包是一个**物流单元**（体积、重量、条码）
- 在零售商眼中：书包是一个**商品**（SKU、进价、售价）

"书包"这个实体，在不同语境下被赋予完全不同的属性、关系和语义角色。没有任何一个单一的本体会话能够自然地同时容纳这四套完全不同的理解框架。

**传统本体论的失察**：它将 `IS-A` 关系视为稳定的、可传递的本体事实，而非视角依赖的解释性产物。语义网的 `owl:Thing` 继承了这种哲学立场，认为世界的概念结构是客观存在的，可以被完整地建模和发现。

**RFD 的修正**：RFD 的多视图本体观认为，同一个软件实体，在不同角色的眼中是不同的事物——对于测试工程师，它是一组可验证的行为；对于运维工程师，它是可部署的运行单元；对于产品经理，它是可追踪的需求实现。RFD 不是要发明一个"超级本体"来统一所有视角，而是承认这些视角的**不可通约性（incommensurability）**，通过多视图的等价映射而非单一全局本体来解决。

#### 3.6.2 维度二：命名多样性与语义等价（Milingual Naming）

**传统本体论的一个未言明的假设**：概念与命名之间存在相对稳定的对应关系。`IS-A` 关系的一端是"概念"（如"书包"），另一端是"类名"（如 `Bag`），二者之间的语义距离是有限的。

**现实中的认知现象**：同一个概念，在不同知识背景、不同业务语境的人口中，有着完全不同的命名：

- 开发者叫它 `backpack`
- 业务分析员叫它 `student-bag`
- 供应链经理叫它 `PPE-2026-Q1`
- 电商系统叫它 `SKU-88512004`

这些命名之间语义等价，但字面上完全不同。"语义等价"在这里的意思是：它们指向同一个软件实体，可以互换使用而不改变系统的任何行为。

**传统语义网本体论的盲点**：语义网（RDF/OWL）通过 URI 命名解决了"全局唯一标识"的问题，但 URI 本身不是人类可读的语义载体。当两个不同的本体系统（比如电商系统和供应链系统）需要集成时，它们会发现彼此对同一实体使用了完全不同的 URI——没有内在机制告诉系统这两个 URI 应该被认定为等价。

**RFD 的多视图愿景中的解决方案**：各视图之间存在语义等价映射。比如，`需求维护视图`中的 `RFC-0004` 与 `代码视图`中的 `ModulePostCalculator` 指向同一个本体节点——它们是同一个事物的不同投影。实现这一等价映射需要的不是更强的 URI 规范，而是视图间的**语义桥接规则**。

### 3.7 软件本体的多视图架构（未来方向）

基于 3.6 节的分析，RFD 的多视图本体愿景与传统语义网本体有本质区别：

```
传统本体论（语义网）：
[客观存在的世界] → 本体建模 → 唯一正确的类层次结构
  问题：视角依赖被忽视；命名多样性无法处理

语义网增强（RFD 愿景）：
[同一本体，多视角投影]
        ┌───────────────────────┼───────────────────────┐
        ▼                       ▼                       ▼
┌───────────────┐  ┌─────────────────┐  ┌──────────────────┐
│   代码视图    │  │  架构设计视图   │  │  需求维护视图     │
│ (Code View)   │  │ (Architecture   │  │ (Requirements    │
│ JAR/WAR产物   │  │  View)          │  │  View)           │
```

**三个核心问题**：

1. **视角边界的确定**：谁来决定哪些属性属于哪个视图？视图的划分是主观的还是客观的？当一个属性同时属于多个视图时，如何处理？

2. **等价映射的形式化**：如何形式化地描述"代码视图中的 `ModulePostCalculator`"和"需求维护视图中的 RFC-0004"是同一个本体节点？是否需要引入"本体等价断言"这样的新概念？

3. **运行时代码包的本体定位**：编译后的 JAR/WAR 是源代码本体的投影终点，还是一个新的独立本体？它与源代码本体之间的关系是什么？

这三个问题目前均无成熟答案，是 RFD 多视图本体愿景的**开放研究问题**，也是它与传统本体论的根本区别所在——RFD 不是宣称已经解决了这些问题，而是把这些未解问题显式地纳入方法论框架，作为未来建模工作的方向。

## 4. 优缺点对比

### 4.1 RFC 驱动开发（RFD）

#### 优点

**1. 认知匹配：捕捉闪现而非等待完整输出**
RFD 是唯一一个从人类认知特性出发的 AI 开发方法论。它不要求用户在对话开始前就完整想清楚所有细节，而是通过 AI 主动提问，在用户的想法闪现的当下将其捕捉并结构化。这显著降低了"用户说不清楚"的沟通成本。

**2. AI 主动提问机制（业界首创）**
AI 在 RFC 生成阶段高主动性参与，主动识别设计模糊点和技术决策点，逐一向用户确认。这不是传统需求访谈的单向过程，而是双向的、迭代的结对式对话。

**3. 代码与 RFC 同等权威，消除"规范 vs 代码"的取舍焦虑**
RFD 不把 RFC 视为"可以被代码覆盖"的文档，而是将代码、语义和 RFC 视为同一设计的三个等价表达。两者的关系是互相验证，而非上下级。

**4. 强制测试用例作为验收标准**
RFC 模板中强制要求"验收准则"章节包含可执行的测试用例（使用注解驱动方式），确保每个设计决策都有可验证的衡量标准。TDD 在 RFC 约束下执行，测试方向由 RFC 定义，避免了"用正确的测试验证错误的设计"的问题。

**5. 自动化提交流水线**
`rfc-driven-development` skill 将"创建分支→提交→推送"自动化，减少了开发者在 Git 操作上的心智负担。

**6. 多视角本体的长期愿景**
RFD 率先提出了"软件作为多视角本体"的概念，并识别出传统本体论的三个根本局限——视角依赖性、命名多样性和等价映射问题。这为需求、架构、代码的统一管理提供了理论框架（尽管尚未完全实现）。

#### 缺点

**1. 缺乏强制质量门控**
代码实现完成后没有强制验证阶段。测试可以写，但提交前没有强制要求所有测试通过。

**2. 评审机制的盲点**
AI 评审使用的是同一模型实例，存在 Cooperative Bias。

**3. RFC 审核者要求较高**
RFC 的质量同时取决于用户的业务理解和架构知识，对审核者的要求较高。对于架构知识不足的审核者，可能无法发现 RFC 中潜在的架构冲突。

**4. 适用范围有限**
RFD 设计上最适合单个功能模块的开发。对于跨模块重构、多团队协作、需要并行探索的场景，缺乏系统性支持。

**5. 缺乏隔离机制**
没有像 Superpowers 那样的 git worktree 隔离机制。

**6. 多视图本体尚未建模**
软件作为多视图本体的愿景目前只是概念阶段，视角边界、等价映射形式化和运行时代码包定位问题尚未解决。

### 4.2 Spec-Driven Development（SDD）

#### 优点

**1. 哲学清晰，代码可丢弃**
"规范是真相，代码是一次性产物"这一哲学帮助团队建立正确的 AI 协作心智模型。

**2. 反馈循环极快**
将传统规范流程（数月周期）压缩到分钟级别。

**3. 三级采用灵活性**
提供三级承诺级别，渐进采用，降低落地门槛。

#### 缺点

**1. 规范质量高度依赖用户**
"Specify" 阶段是用户驱动而非 AI 引导的。

**2. 认知模型错位**
SDD 假设人类能够线性输出完整规范，这与人类思维的真实特性不符。

**3. 无正式质量门控**
没有强制测试要求，没有强制评审。

### 4.3 Verified Spec-Driven Development（VSDD）

#### 优点

**1. 最强的质量保障体系**
六阶段流水线，每一阶段都有明确的产出标准和验证要求。

**2. 角色分离设计合理**
Architect（人类）、Builder（AI）、Tracker（Issue 系统）、Adversary（批评 AI）四个角色分工合理。

**3. 可裁剪**
Lean Mode 用 20% 的努力获得 80% 的 VSDD 价值。

#### 缺点

**1. 流程过重**
六阶段流水线对小型功能开发过于繁重。

**2. 规范错误时 TDD 方向也会错**
Spec Crystallization 在 Plan 之前，规范有问题时 TDD 测试也会是错误方向。

### 4.4 Superpowers

#### 优点

**1. 强制人机协作纪律**
确保人类始终是最终决策者。

**2. git worktree 隔离**
避免实验性代码污染主分支。

**3. 成熟度高**
139k+ GitHub stars。

#### 缺点

**1. 交互频率过高**
过多的暂停点会打断有经验开发者的心流。

**2. AI 主动性被压制**
Superpowers 的设计哲学是"让 AI 停下来等人"，与 RFD 的"让 AI 主动引导用户"形成了有趣的对比。

**3. 对抗性评审缺失**
Review 阶段使用同一个 AI 实例，存在合作偏差。

## 5. 本质差异总结

### 5.1 核心差异矩阵

| 维度 | RFD（自研） | SDD/VSDD | Superpowers |
|------|------------|----------|-------------|
| **出发点** | 人类认知特性（闪现式思维） | 代码作为产物的可替代性 | 人机协作纪律 |
| **规范vs代码的权威** | 代码、语义、RFC 三者同等 | 规范是真相，代码是产物 | 人 > 规范 > AI |
| **人机交互节奏** | 规范阶段深度交互，开发阶段低频 | 规划阶段一次性交互 | 每任务前高频审批 |
| **AI 主动性** | 极高（主动提问捕捉闪现） | 低（被动响应规范） | 低（等待审批） |
| **TDD 的位置** | 质量保障核心（在 RFC 约束下增强） | 增强版 TDD（VSDD 中强制） | TDD 强制 |
| **文档可变性** | RFC 生成后通常不改 | 规范可随实现更新 | 无明确立场 |
| **多视图支持** | 规划中（多视角本体，含开放研究问题） | 无 | 无 |
| **隔离机制** | 无 | 无 | git worktree 隔离 |
| **适用规模** | 中小型功能 | 中小型（SDD）/ 高保证（VSDD） | 中大型项目 |

### 5.2 哲学层面的根本分歧

四种路线的本质分歧在于：**"AI 与人类认知如何协作？人类思维的真实特性是什么？"**

- **RFD 路线**：人类思维是**闪现的、碎片化的、有限的**。AI 通过主动提问在想法闪现的当下捕捉并结构化。代码、语义、RFC 是同一软件本体的三个等价视图，同等权威。软件是可以从多个视角理解的多视图本体，传统本体论的视角依赖性和命名多样性盲点需要被显式建模。
- **SDD/VSDD 路线**：人类能够**线性地输出完整规范**，AI 在规范约束下生成代码。
- **Superpowers 路线**：人类是**全程审批者**，AI 的每一步都需要人类确认。

## 6. 命名建议

### 6.1 方法论命名分析

当前这个方法论已经远超"RFC 驱动开发"所能涵盖的范畴。"RFC 驱动"这个名字暗示 RFC 只是一个开发依据，而实际上这个方法论的核心创新在于：

1. **认知匹配的 RFC 生成**——AI 主动捕捉人类闪现的想法并结构化
2. **三方等价的本体观**——代码、语义、RFC 语义等价，不是上下级关系
3. **多视角软件本体愿景**——软件是可以从多个视角理解的统一本体
4. **TDD 在规范约束下执行**——测试方向由 RFC 定义，不是开发者自定

### 6.2 候选名称

| 候选名 | 全称 | 含义 | 优点 | 缺点 |
|--------|------|------|------|------|
| **EDD** | Epiphany-Driven Development（顿悟驱动开发） | Epiphany = 顿悟、闪现 | 精准捕捉认知科学切入点和多视角本体的涌现特性 | 非英语母语者不够直观 |
| FDD | Flash-Driven Development（闪现驱动开发） | Flash = 闪光、闪现 | 简洁直观 | FDD 缩写与 Feature-Driven Development 冲突 |
| ODD | Ontology-Driven Development（本体驱动开发） | Ontology = 本体论 | 精准描述多视图本体愿景 | 过于学术；"本体驱动"容易被误解为语义网 |
| CDD | Crystallize-Driven Development（晶化驱动开发） | Crystallize = 使结晶 | 隐喻优雅，暗示 RFC 是想法的自然凝固 | 较抽象 |
| IDD | Insight-Driven Development（洞见驱动开发） | Insight = 洞见 | 易懂 | 含义宽泛 |
| SDD | Spark-Driven Development（火花驱动开发） | Spark = 火花 | 形象生动 | SDD 缩写与 Spec-Driven Development 冲突 |

### 6.3 最终推荐

**推荐名称：Epiphany-Driven Development（EDD）—— 顿悟驱动开发**

核心理由：

1. `Epiphany`（顿悟）精确捕捉了人类认知的真实特性——设计不是线性推导的，而是从碎片中涌现的顿悟。符合 RFD 出发点（闪现式思维）和发展愿景（多视角本体的涌现结构）
2. AI 的角色是**捕捉和结构化**顿悟，而非代替人类产生顿悟——这个名字准确描述了人机分工
3. 可以缩写为 `EDD`，与 `RFD`（RFC-Driven Development）同风格，但不冲突
4. 内涵丰富但不晦涩：既指向认知科学的顿悟理论，又暗含"结构化"的过程（顿悟需要被记录才能传承），还隐喻了多视角本体的涌现性（顿悟不是推导出来的，而是从碎片中自然涌现的）

## 7. 相互可以借鉴的点

### 7.1 EDD 可以从 VSDD 借鉴的

**1. 引入对抗性评审（Adversarial Review）**
当用户要求评审 RFC 时，启动一个新的 AI 会话（无历史上下文）专门挑刺，而不是在当前对话中让同一个 AI 评审自己写的 RFC。

**2. 增加强制质量门控**
增加一个强制门控：所有测试通过之前不允许提交。明确定义"测试全部通过"作为提交的前置条件。

**3. 增加工作隔离机制**
对于探索性开发，使用 git worktree 创建隔离的工作环境。

### 7.2 EDD 可以从 Superpowers 借鉴的

**1. 任务级的审批机制**
在 RFC 生成后、进入开发阶段前，增加一个"开发计划审批"步骤：AI 根据 RFC 生成具体的实现任务列表，用户逐项确认后再开始开发。

**2. 更细粒度的进度追踪**
增加进度状态追踪，让用户清楚看到当前处于哪个阶段。

### 7.3 EDD 可以从 SDD 借鉴的

**1. Plan 阶段的强制技术约束**
SDD 的 Plan 阶段强制要求定义技术约束、栈决策和架构模式。EDD 的 RFC 模板可以更明确地要求填写技术选型的替代方案和选择理由。

**2. 规范的持久性承诺**
若开发中发现 RFC 有重大遗漏，应优先回到 RFC 生成阶段补充，而非在代码中绕过 RFC。

### 7.4 业界方法论可以借鉴 EDD 的

**1. AI 主动提问机制（最具输出价值）**
EDD 最值得向业界输出的创新点：AI 在需求理解阶段**主动提问**来捕捉人类闪现的想法。VSDD 和 Superpowers 都假设人类已经想清楚了需求，而 EDD 让 AI 帮助人类想清楚。

**2. 代码与规范同等权威的立场**
EDD 将代码、语义和 RFC 视为同等权威——这是对 SDD"代码是产物"的实践性改进。

**3. 强制测试用例作为 RFC 验收准则**
EDD 的 RFC 模板要求验收准则必须包含可执行的测试代码。可以借鉴到 VSDD 的 Spec Crystallization 阶段。

**4. 软件本体多视角的愿景**
EDD 率先提出了"软件作为多视角本体"的概念，并识别出传统本体论在视角依赖性和命名多样性上的根本缺失。这为整个行业提供了一种新的软件组织思路。

## 8. 结论

**命名为 Epiphany-Driven Development（EDD）—— 顿悟驱动开发**。

EDD 与 SDD/VSDD/Superpowers 相比，有五个层面的核心差异：

**第一，出发点的差异。** EDD 是唯一一个从人类认知科学出发的 AI 开发方法论。它承认人类思维是闪现的、碎片化的、有限的，并以此为方法论设计的起点。SDD 和 Superpowers 都隐含假设人类能够线性输出完整需求或分解清晰任务。

**第二，RFC 生成依赖条件的明确性。** EDD 明确指出 RFC 的生成依赖三大输入：CORE-DESIGN.md 和 ACCEPTANCE.md 提供的架构约束空间、用户原始零星的闪现想法、以及迭代式初稿反馈循环。这使得 RFC 生成过程有据可依。

**第三，人类双重角色的显式定义。** EDD 显式定义了人类在 RFC 审核中的双重角色：业务意图的判断者 + 架构可行性的审核者。

**第四，规范权威性的差异。** EDD 将代码、语义和 RFC 视为同一软件本体的三个等价表达，三者同等权威。

**第五，多视角软件本体的长期愿景。** EDD 率先识别出传统本体论的三个根本局限：视角依赖性被忽视、命名多样性无法处理、等价映射未形式化。EDD 的多视图本体愿景不是宣称已经解决了这些问题，而是把这些未解问题显式纳入方法论框架，作为未来建模工作的方向。

对于 jmixconfigengine 项目，EDD 本身已经是最匹配的方法论。建议优先引入：① 对抗性评审（用新会话做评审）② 强制测试门控（所有测试通过前不可提交）③ git worktree 隔离。多视角本体的建模可以作为长期研究方向。

## 9. 哲学基础：EDD 的思想谱系与理论支撑

> 本章节专门介绍 EDD 方法论背后的哲学思想来源，解释这些哲学观点的含义，以及它们如何支撑 EDD 方法论的各个设计决策。

### 9.1 尼采的视角主义（Perspectivism）

**人物与著作**：弗里德里希·尼采（Friedrich Nietzsche），19 世纪末德国哲学家，代表作《论道德的谱系》（On the Genealogy of Morals，1887）。

**核心观点**：尼采是哲学史上第一个系统阐述"视角主义"的哲学家。他的核心论断是：

> "事物的本质始终是'视角性的'（perspectival），这只是我们自身条件的一部分。"（《论道德的谱系》，第三论文第 12 节）

尼采认为，不存在"纯粹的客观知识"这回事——所有知识的产生都受到观察者视角的塑造。但这并不意味着知识是任意的或相对主义的；相反，尼采主张通过**积累多重视角**，我们可以获得比任何单一视角更丰富的理解。不同视角不是要被裁决哪个正确，而是要被编织在一起，形成更全面的图景。

**对 EDD 的支撑**：

EDD 的多视图本体愿景直接承袭了尼采的视角主义。在 EDD 看来，同一个软件实体（"书包"），在测试工程师眼中是一个可验证的行为集合，在运维工程师眼中是一个可部署的运行单元，在产品经理眼中是一个可追踪的需求实现——这些视角都是合法的，没有哪个视角是"更真实"的。EDD 的方法不是要在这些视角之间裁决出一个"正确"的本体重构，而是通过语义等价映射，将它们编织成一个多视图的软件本体。

### 9.2 海德格尔的存在论现象学（Phenomenological Ontology）

**人物与著作**：马丁·海德格尔（Martin Heidegger），20 世纪最重要的德国哲学家之一，代表作《存在与时间》（Being and Time，1927）。

**核心观点**：海德格尔提出了一个根本性的哲学追问："存在（Being）究竟是什么意思？"他区分了两个概念：

- **存在者（Seiende / beings）**：具体的、可以被经验到的事物（如一本书、一段代码）
- **存在（Sein / Being）**：使存在者得以存在的那个"是"本身——这不是一个事物，而是一种状态或发生

海德格尔进一步指出，人类理解"存在"的方式，不是通过冷静的观察，而是通过**在世存在（Being-in-the-world）**——人类从来不是作为中立的观察者面对世界，而是始终已经卷入世界、与世界打交道（打交道 = 使用工具、关切事物、担忧未来）。这种卷入的状态，海德格尔称之为 **"操心"（Sorge / Care）**。

他著名的"锤子案例"（the hammer example）说明：当我们正常使用一把锤子时，锤子不是作为一个"对象"被我们认知——它是"透明的"，我们直接通过它完成目的。只有当锤子坏了（不可用），我们才突然意识到它的存在，把它当作一个对象来分析。

**对 EDD 的支撑**：

海德格尔的现象学为 EDD 的两个设计决策提供了哲学依据：

1. **闪现式思维的合理性**：海德格尔认为人类对世界的理解从来不是"中性观察"，而是在"操心"中展开——当开发者在深夜突然想到一个设计思路，这个闪现不是理性的偏差，恰恰是人类在世存在的自然方式。EDD 的 AI 主动提问机制，正是要捕捉这些在"操心"中涌现的闪现想法。

2. **代码与 RFC 的同等权威**：海德格尔区分了"存在者"（代码、RFC 文档）与"存在"（设计意图的语义）。传统方法论把"存在者"中的某一个（通常是规范文档）提升为权威；EDD 则认为两者都是设计意图这个"存在"的投影——代码不是"比 RFC 更真实"的产物，RFC 也不是"比代码更抽象"的真相。

### 9.3 卡尔·波普尔的三个世界（World 1, 2, 3）

**人物与著作**：卡尔·波普尔（Karl Popper，1902-1994），奥地利裔英国哲学家，代表作《客观知识》（Objective Knowledge，1972）。

**核心观点**：波普尔在 1972 年的演讲中提出了著名的"三个世界"理论：

- **世界 1（World 1）**：物理世界——物理对象、物质、能量
- **世界 2（World 2）**：精神世界——意识、主观经验、信念
- **世界 3（World 3）**：客观知识世界——语言、理论、书、图书馆、论证；人类思想的客观产物

波普尔的创新在于：世界 3 是**客观的**，不是因为它是"正确的"或"最终的"，而是因为它的存在**独立于任何单一的主观意识**。一本写好的书，其意义不取决于读者怎么想；一个已经被证明是错误的科学理论，它仍然存在于世界 3 中——可以被批评、被引用、被研究。

同时，波普尔强调世界 3 与世界 2（人类意识）之间存在**反馈回路**：人类通过思考世界 3 中的思想产物，创造新的世界 3 思想，而这些新思想反过来又改变了人类的世界 2 意识。

**对 EDD 的支撑**：

波普尔的三个世界为 EDD 的多视图本体提供了精确的分类框架：

```
世界 1（物理世界）
└── 运行时代码包（JAR/WAR）：编译后的物理产物

世界 2（精神世界）
└── 开发者的理解、闪现的想法、业务意图
    └── 这是 EDD 的起点：捕捉闪现的想法（世界 2）

世界 3（客观知识世界）
├── 代码本体（Java 类、方法的客观结构）
├── RFC 文档（文本化的设计知识）
└── 架构设计文档（CORE-DESIGN.md）
    └── 这些都是世界 3 的产物，彼此语义等价
```

波普尔的"反馈回路"精确描述了 EDD 的工作流程：人类在世界 2 中的闪现想法（Phase 1）→ 被结构化为 RFC（世界 3 产物）→ RFC 驱动代码开发（世界 3 到世界 1）→ 代码运行后的行为反过来更新人类对设计的理解（世界 1 到世界 2 的反馈）。

### 9.4 具身认知与概念隐喻理论（Lakoff & Johnson）

**人物与著作**：乔治·莱考夫（George Lakoff）和马克·约翰逊（Mark Johnson），美国认知语言学家，合著《我们赖以生存的隐喻》（Metaphors We Live By，1980）。

**核心观点**：

莱考夫和约翰逊提出了两个革命性的观点：

**第一，抽象概念是具身的（Embodied）**。传统观点认为抽象概念（如"自由"、"正义"、"软件架构"）是脱离身体的纯粹理性产物。他们的研究表明：所有抽象概念都是通过**概念隐喻**从身体经验中构建出来的。比如，"理解一个观点"在英语中被表达为"抓住一个观点"（grasp an idea）——这并非偶然，而是因为人类最早通过触觉（抓住物体）来理解"掌握"这个概念，然后将其隐喻性地映射到认知领域。

**第二，隐喻是思维本身的结构（Metaphors We Live By）**。他们发现，人类日常语言中充满了这样的概念隐喻：
- 理解 = 看见（"I see what you mean"）
- 时间 = 资源（"spending time"，"wasting time"）
- 想法 = 物体（"That's a good idea"，"I can't grasp that concept"）
- 软件 = 建筑（"architecture"，"foundation"，"building on top of"）

这些隐喻不是修辞装饰，而是思维本身的结构——我们无法不使用隐喻来思考抽象概念。

**对 EDD 的支撑**：

莱考夫和约翰逊的理论为 EDD 的闪现式思维提供了认知科学依据：

1. **闪现式思维的认知合理性**：当开发者脑海中突然涌现一个设计想法（"这应该像水管一样，弯的地方要有阀门"），这不是理性的偏差，而是一个**具身隐喻**在起作用——开发者正在将"流体管道"的物理经验，隐喻性地映射到"数据流控制"的抽象领域。EDD 的 AI 主动提问机制，正是要帮助开发者显式表达这些隐喻性的闪现想法，并将它们结构化为 RFC 中的明确设计。

2. **软件多视图本体的概念隐喻结构**：在 EDD 的多视图本体中，"代码视图"、"架构视图"、"需求视图"之间的关系，本身就是一组概念隐喻——我们将同一个软件实体从不同角度"看见"，就如同我们从不同角度"看见"一个雕塑。没有任何单一视角能穷尽雕塑的全部信息，也没有哪个视图能穷尽软件本体的全部语义。

### 9.5 哲学基础总览与 EDD 方法论的映射

| 哲学思想 | 核心主张 | 对 EDD 的支撑 |
|---------|---------|-------------|
| **尼采视角主义** | 不存在纯粹的客观知识，所有知识都是视角性的；通过积累多视角获得更全面的理解 | 多视图本体的哲学基础——同一软件实体在不同角色视角下是不同的事物 |
| **海德格尔存在论** | 人类通过"在世存在"理解世界，理解不是观察而是卷入；闪现的想法是人类在操心中的自然涌现 | AI 主动提问机制的合理性；代码与 RFC 的同等权威（非上下级） |
| **波普尔三个世界** | 世界 3（客观知识）是独立于单一意识存在的思想产物；世界 2 与世界 3 存在反馈回路 | 多视图本体的分类框架（世界 1=运行产物，世界 2=闪现想法，世界 3=代码+RFC+架构） |
| **莱考夫具身认知** | 抽象概念通过概念隐喻从身体经验构建；隐喻是思维本身的结构而非装饰 | 闪现式思维的认知科学依据；多视图的隐喻性（从不同角度"看见"同一个本体） |
