# RFC-0007 复盘：Validator 规则类型兼容问题

## 背景

RFC-0007 的目标不是只新增一种结构化规则的局部能力，而是把结构化规则纳入现有规则体系：规则可以用于生成，也可以用于校验；规则形态可以是 Struct 结构化规则，也可以是 Code Rule，后续还可能继续扩展。

这次修正中暴露出的核心问题是：新增的 `Validator` 接口最初只对 Struct 规则有效，不能覆盖 Code Rule-only、Struct + Code Rule 混合等场景。这说明实现偏离了 RFC 中“结构化规则应转换成约束表达式，并进入统一约束模型”的设计方向。

## 现象

最初实现把 `validate(ModuleInst)` 理解成“读取结构化规则并手工判断实例是否满足”，导致校验逻辑与具体规则类型绑定。

这带来几个直接问题：

1. 只有 Struct 规则能被校验。
2. Code Rule-only 场景无法通过同一个 Validator 生效。
3. Struct + Code Rule 混合场景无法保证整体规则一致性。
4. 校验路径和生成路径没有复用统一的求解模型，导致同一批规则在不同入口下语义可能不一致。

## 根因判断

这个问题不是单纯发生在 RFC 阶段，也不是单纯发生在编码阶段。更准确地说，是 RFC 没有把关键设计约束写成不可误解的验收条件，编码阶段又用局部可工作的方式绕过了系统已有抽象。

RFC 阶段的问题在于：文档提到了“结构化规则转换成类似约束表达式的形式”，但没有把它提升为硬性设计不变量，例如：

> Validator 不能解释某一种具体规则类型；Validator 只能把待校验实例转换成固定输入约束，然后复用统一的 CP/rule 模型完成校验。

编码阶段的问题在于：实现时没有先确认系统已有的规则执行抽象，而是新增了一条 Struct 专用判断路径。这个路径在 Struct-only happy path 下看起来能工作，但它破坏了规则体系的统一性。

测试阶段的问题在于：测试只证明了“Struct 规则可以被校验”，没有证明“Validator 对所有规则类型通用”。缺少 Code Rule-only、Struct + Code Rule 混合、生成与校验路径一致性这些反例场景。

## 关键设计不变量

后续类似功能必须先明确这些不变量：

1. Public API 不应该绑定具体规则类型。
2. `validate(ModuleInst)` 的职责是校验一个给定实例，而不是补全一个可行解。
3. 结构化规则、Code Rule 和未来规则都应进入统一规则构建和求解模型。
4. 结构化规则用于生成时仍应走原有 `inferParas` / InfoParse 风格入口。
5. 结构化规则用于校验时应转换为约束模型的一部分，而不是在 Validator 中手写规则解释器。
6. 诊断结果应面向业务规则，不应把输入固定约束、系统约束这类内部约束当作业务冲突返回。

## RFC 阶段应改进的写法

以后 RFC 中只写“支持某能力”是不够的，还要写清楚 public API 的语义契约。

例如 `validate(ModuleInst)` 应明确：

1. 输入是一个已经给定的模块实例。
2. 已出现的 part 数量按实例固定。
3. 未出现的可选 part 在校验中应视为数量 `0`。
4. 校验不允许求解器自动补齐实例来使规则通过。
5. 校验应复用统一规则模型，而不是按规则类型分支解释。
6. invalid 时返回的 rule code 应是业务规则 code。

RFC 还应明确反例和禁止做法：

1. 禁止在 `ModuleConstraintExecutor.validate` 这类 public boundary 中出现面向具体规则 schema 的 `instanceof` 分支。
2. 禁止只为 Struct 规则新增独立校验器，绕过 Code Rule 和统一 CP 模型。
3. 禁止只覆盖 Struct-only 测试后就认为 Validator 已完成。

## 代码阶段应改进的检查点

实现前应先做一次“系统抽象定位”：

1. 找到现有规则从声明到执行的完整链路。
2. 找到生成路径和校验路径是否已经有可复用入口。
3. 判断新增能力应该接入哪一层，而不是先写局部逻辑。
4. 检查 public API 是否出现对内部规则类型的直接依赖。

如果发现新增逻辑需要识别具体规则类型，优先判断这是否应该下沉到 rule build / rule expand / rule execution 层，而不是留在 executor 的 public 方法里。

## 测试阶段应补齐的验收矩阵

每个规则能力至少要覆盖下面这些组合：

| 场景 | 生成 | 校验通过 | 校验失败 |
| --- | --- | --- | --- |
| Struct-only | 必测 | 必测 | 必测 |
| Code Rule-only | 按能力必测 | 必测 | 必测 |
| Struct + Code Rule 混合 | 必测 | 必测 | 必测 |
| 黑名单 / 白名单 / 组合限制 | 按 RFC 必测 | 必测 | 必测 |
| 未来新增规则类型 | 应能复用同一 Validator 语义 | 应能扩展 | 应能扩展 |

同时要覆盖路径一致性：

1. 同一批规则用于 `inferParas` 生成时结果正确。
2. 同一批规则用于 `validate(ModuleInst)` 校验时结果正确。
3. 一个由生成路径得到的合法实例，应能被校验路径判定为合法。
4. 一个违反任意业务规则的实例，应能被校验路径判定为非法。

## 后续执行准则

后续做 RFC 驱动开发时，必须把下面几项作为开发前检查：

1. **语义契约先行**：每个新增 public API 都要写输入、输出、副作用、边界语义和失败语义。
2. **验收矩阵先行**：不要只写主路径测试，要先列出规则类型组合和正反例。
3. **抽象归属先行**：判断逻辑属于产品基础数据生成、规则构建、求解执行、校验入口还是测试 helper。
4. **反例先行**：至少写一个“如果按错误方式实现会失败”的测试场景。
5. **路径一致性检查**：如果一个能力同时用于生成和校验，必须同时覆盖两个入口。

## 结论

这次问题的根本原因，是设计意图没有被固化成足够明确的工程约束和验收矩阵，导致编码时选择了局部可工作的实现。

以后 RFC 不只要说明“要做什么”，还要说明“这个能力必须接入哪个抽象层”“哪些实现方式是禁止的”“哪些规则组合必须通过测试”。这样才能在写代码前发现偏差，而不是等到调试 Validator 时才发现规则体系已经被局部实现绕开。
