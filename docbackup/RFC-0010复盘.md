# RFC-0010 复盘

> 日期：2026-06-05
> 主题：如何尽量一次性生成 RFC，并尽量一次性根据 RFC 生成代码

## 复盘结论

这次回顾的重点，确实应该是“发现还缺失哪些知识”，但不只是补知识库。更准确地说，是把三类内容沉淀下来：

1. 项目事实：当前框架已经有什么能力、惯用入口、隐含规则。
2. 用户偏好：哪些写法是“简洁、自然、符合框架精神”的。
3. 流程护栏：AI 在生成 RFC 和写代码前，必须做哪些核对，才能减少返工。

目标不是让 AI 凭空一次性猜对，而是让 AI 在动手前先查到足够多项目事实。

## 暴露出的缺失知识

### 1. 测试框架既有能力没有优先检索

最初额外封装了 `singleReqs`、`crossTotal`、`hasSolutionWithSelectedParts`，但当前测试框架已经可以直接使用 `inferRecommendModule(...)` 和 DSL 写法。

缺失知识：

- `ModuleScenarioTestBase` 已经提供了高层测试入口。
- RFC 和实现前必须先找相似测试，例如 `DynMultReq4MultiReqTest`。
- 如果现有基类和 DSL 能表达，就不应在测试类里新增局部请求对象拼装 helper。

### 2. 测试 DSL 能表达跨分类语义这一点没有进入 RFC 样例

后来明确的写法是：

```text
disk:Sum_Quantity ==1 where Capacity=40
disk,mainBoard:Sum_Capacity >=100 where PortRate=10G
```

这比新增请求 helper 更贴近当前框架。

缺失知识：

- 测试 DSL 的 grammar 和表达惯例需要被 skill 显式收录。
- 单分类用 `xxx:`。
- 总体约束用 category code 列表，例如 `xxx1,xxx2:`。

### 3. “不要额外封装”的项目偏好没有变成约束

RFC 虽然写了“不新增中间 DTO”，但测试建议里又出现了 `totalReq(...)` 这类 helper，和实际偏好冲突。

缺失知识：

- RFC 的测试封装建议需要与现有测试基类能力对齐。
- 如果已有 DSL 和 helper 足够表达，RFC 不应再建议新 helper。
- 新 helper 必须说明为什么现有基类无法覆盖。

### 4. 字段可推导原则没有贯彻到底

`CrossCategoryPartCategoryConstraintReq.code` 后来被删掉，因为可以由 `categoryCodes` 拼接得到。

缺失知识：

- 不仅 RFC 要写“避免重复配置”，实现时也要逐字段审查。
- 每个新增字段都要回答：这是业务输入，还是可以从上下文稳定推导。
- 如果字段只是内部唯一标记，应优先从既有输入派生。

### 5. category code 匹配语义没有提前澄清

最初构建器用了 prefix/多实例风格查找，后来调整为按 category code 精确匹配，每个 code 获取一个 `PartCategoryAlgImpl`。

缺失知识：

- 运行时 alg 查找的语义边界需要文档化。
- 什么时候使用精确 category code 查找。
- 什么时候才需要 prefix 或实例展开查找。
- 逻辑 category、多实例实例、过滤后 alg 之间的关系需要更清楚。

### 6. 测试 short code 规则没有被主动发现

`Part.code.length() <= 4` 时会直接作为 short code，否则生成 `PTn`。这影响断言可读性。

缺失知识：

- 测试断言里的 short code 生成规则需要沉淀到测试知识库。
- 测试样例应尽量使用短 code，例如 `d1`、`b1`、`e1`。
- 这样断言可以保留语义，而不是依赖 `PT1`、`PT2` 这类自动编号。

### 7. 注解生成规则没有前置确认

`@PartAnno.code` 可以由字段名推导，很多 `code = "xxx"` 是冗余信息。

缺失知识：

- `ModuleGenneratorByAnno` 的字段名推导规则应进入“建模注解惯例”知识。
- 测试样例中能由字段名推导的 code 不要重复写。
- `fatherCode`、`required`、`supportMultiInst` 这类真正表达语义的字段才保留。

### 8. 测试样例没有遵守“只表达本测试关注点”

例如 `price` 没被用到，却写进了样例。长 code 也让断言失去语义。

缺失知识：

- RFC 测试样例要有“最小语义样例”准则。
- 不用的属性不写。
- 断言需要什么，就围绕断言设计什么。
- 测试代码应该让读者一眼看到业务语义。

### 9. 调试习惯没有进入验收模板

实际希望每个测试都调用 `printSimpleSolutions()`，方便调试看所有解。

缺失知识：

- 推荐/求解类测试默认打印 simple solutions。
- 失败分支也可以打印，输出 `No solutions found`，不影响断言。

### 10. RFC 的 P0 范围和代码实现路线有局部张力

RFC 里写了“多实例逻辑分类所有实例参与汇总”，后续又调整为精确 category code 查询。这说明 RFC 对“输入 category code 到运行时实例”的映射语义还不够清楚。

缺失知识：

- 需要把逻辑 category、多实例实例、过滤后 alg、精确 code 查找的关系讲清楚。
- P0 到底支持哪一层语义，不能只在伪代码里模糊表达。

## Skill 改进方向

### 1. 改进 `interactive-rfc-generation`：生成 RFC 前增加代码库侦察阶段

在写 RFC 初稿前，不只是读 `CORE-DESIGN.md` 和 `ACCEPTANCE.md`，还要强制查：

- 相似测试类。
- 测试基类 helper。
- 现有 DSL/解析器能力。
- 注解生成惯例。
- 相关实现类的现有入口。

### 2. 改进 `interactive-rfc-generation`：RFC 必须包含复用优先清单

每个 RFC 增加一节：

- 优先复用哪些现有类/方法。
- 不新增哪些 helper/DTO/字段。
- 哪些信息由字段名、category list、表达式或上下文推导。

### 3. 改进 `interactive-rfc-generation`：测试样例必须来自现有框架写法

RFC 里的测试代码不能凭空设计 helper。要先引用一个相似测试来源，例如：

- `DynMultReq4MultiReqTest`
- `MultiPCTest`
- `OptionalPartCategoryWhitelistGuardTest`

### 4. 改进 `interactive-rfc-generation`：增加样例简洁性检查

RFC 测试样例生成后检查：

- 是否有未使用字段，如 `price`。
- part code 是否过长，导致断言只能用 `PTn`。
- 是否能用字段名推导 code。
- 是否展示了 `printSimpleSolutions()`。
- 是否用了项目真实 DSL，而不是临时对象拼装。

### 5. 改进 `rfc-driven-development`：实现前增加最小新增代码挑战

开发前必须回答：

- 这个 helper、字段、DTO 是否必须新增。
- 现有基类或 DSL 能不能表达。
- 这个 code 是业务输入，还是能推导。
- 是否只是为了测试方便而新增生产结构。

### 6. 改进 `rfc-driven-development`：实现前做 RFC 与当前代码偏差审查

RFC 是设计，但代码库会演进。实现前需要核对：

- RFC 示例是否过时。
- RFC 建议的方法名是否真实存在。
- 是否已有更自然的项目入口。
- 是否有后来 RFC 或提交改变了基础语义。

### 7. 改进 `rfc-driven-development`：测试前置 checklist 更具体

现在 skill 已经说“优先复用测试基类”，但需要补成可执行检查：

- 先 `rg inferRecommendModule`。
- 先找 2 个相似测试。
- 禁止在测试类里先写请求对象拼装 helper，除非基类没有能力。
- 断言优先使用语义 short code。
- 推荐/求解类测试默认 `printSimpleSolutions()`。

## 知识结构改进方向

### 1. 新增项目测试惯例索引

内容包括：

- `inferRecommendModule` DSL 示例。
- `assertSoluContain` 使用方式。
- `printSimpleSolutions` 使用习惯。
- short code 生成规则。
- 多实例断言格式，例如 `I1_d2(...)`。

### 2. 新增注解建模惯例索引

内容包括：

- `@PartAnno.code` 什么时候可省。
- 字段名如何推导 code。
- `fatherCode` 什么时候必须写。
- `price`、`attrs`、`required`、`supportMultiInst` 的测试使用原则。

### 3. 新增算法层查找语义索引

内容包括：

- 精确 category code 查找。
- prefix/inst 查找适用场景。
- 逻辑 category 与多实例 category 的关系。
- filtered candidate 与 original candidate 的边界。

### 4. 新增 RFC 实现前知识卡片

每次进入实现前，AI 应先填一张小卡片：

```text
相似测试：
现有 helper：
现有 DSL：
需要新增的字段：
可以推导的字段：
需要新增的生产类：
可能误用的旧逻辑：
回归测试范围：
```

如果这张卡片填不出来，就说明还没到动手实现的时候。

## 生成中校验与生成后校验的平衡

这两类校验不应该二选一，而应该分层：

1. **生成中校验**：负责低成本、确定性、可机械检查的问题。
2. **生成后校验**：负责整体一致性、架构判断、复杂语义和第二视角。
3. **复盘沉淀**：负责把生成后才发现的问题，判断哪些可以前移为下一次的生成中校验。

### 生成中适合校验什么

生成中校验适合做“能快速拦住明显偏差”的事情：

- 是否读取了相似 RFC、相似测试和现有 helper。
- 是否优先复用了 `inferRecommendModule(...)`、`ModuleScenarioTestBase` 等现有入口。
- 是否新增了可由字段名、category list、表达式或上下文推导的字段。
- 是否设计了不必要的测试局部 helper。
- RFC 测试样例是否使用项目真实 DSL。
- part code 是否足够短，便于断言使用语义化 short code。
- 是否写了未使用字段，如 `price`。
- 推荐/求解类测试是否调用 `printSimpleSolutions()`。

这类校验应尽量写进 `interactive-rfc-generation` 和 `rfc-driven-development`，因为它们越早发现，返工成本越低。

### 生成后适合校验什么

生成后校验适合做“需要完整上下文和判断力”的事情：

- RFC 的 P0 边界是否真实可实现。
- RFC 示例是否和当前代码库存在偏差。
- 实现是否过度抽象或过度封装。
- 字段和 DTO 是否在长期模型中合理。
- 测试是否覆盖了真实风险，而不是只覆盖 happy path。
- 是否有新的用户偏好或项目事实需要沉淀。

这类校验适合交给 `rfc-driven-retrospective`，必要时再触发人工评审或独立评审。

### 平衡原则

可以用一句话判断：

```text
能被 checklist 或脚本稳定判断的，前移到生成中；
需要架构判断、语义权衡或第二视角的，放到生成后；
复盘负责把反复出现的生成后问题，升级为下一次的生成中护栏。
```

因此，不断增加 skill 内容不是为了把所有判断都塞进生成阶段，而是为了形成一个质量闭环：

```text
生成中快速护栏
  -> 生成后整体评审
  -> 复盘识别缺失知识
  -> 更新 skill / 文档 / checklist
  -> 下一次更早发现
```

## 当前推荐的 skill 组合

### 1. RFC 生成前/中

使用 `interactive-rfc-generation`。

重点：

- 代码库侦察。
- 复用优先清单。
- 不新增清单。
- 接口矩阵。
- 测试样例约束。
- 用户问题澄清。

目标：让 RFC 初稿就尽量贴近当前代码库，而不是写一个抽象上正确、实现时才发现不顺手的方案。

### 2. 根据 RFC 开发时

使用 `rfc-driven-development`。

重点：

- 实现前知识卡片。
- 最小新增代码挑战。
- 字段可推导审查。
- 测试惯例检查。
- 相关回归测试。

目标：避免“读完 RFC 就直接写”，而是在动手前先确认当前项目已有能力。

### 3. 生成后复盘/流程改进

使用 `rfc-driven-retrospective`。

重点：

- 对照 RFC 预期和实际实现。
- 找出用户纠偏点。
- 识别缺失的项目事实、用户偏好和流程护栏。
- 给出 skill 和知识结构改进方向。

目标：把返工转化为下一次的前置能力。

### 4. 更新或创建 skill

使用 `skill-creator`。

重点：

- 把复盘结论压缩为可触发、可执行、可验证的 skill。
- 不把所有复盘细节塞进 skill；细节可以沉淀到 references 或项目文档。

## 推荐流程

针对本项目这类“领域 DSL + 约束求解 + 注解建模 + RFC 驱动演进”的系统，建议采用以下流程：

```text
1. 需求输入
   -> 明确业务语义、使用入口和 P0 范围

2. 代码库侦察
   -> 查相似 RFC、相似测试、现有 DSL、helper、生成器规则

3. RFC 初稿
   -> 写设计、验收矩阵、复用优先清单、不新增清单

4. 生成中校验
   -> 用 checklist 检查样例、字段、helper、测试风格

5. RFC 评审
   -> 用户确认语义边界、测试表达和实现方向

6. 实现前知识卡片
   -> 确认相似测试、现有 helper、可推导字段、回归范围

7. 代码实现
   -> 小步实现，优先复用现有结构

8. 自动化验证
   -> 新测试 + 相关回归 + 必要时完整测试

9. 生成后复盘
   -> 找出返工原因，分类为项目事实、用户偏好、流程护栏

10. 知识刷新
    -> 更新 skill、项目知识文档、测试惯例索引
```

这个流程本质上是把“质量”前移，但仍保留生成后的独立评审。前移解决确定性错误，后评审解决复杂判断。

## 可参考的业界流程

本项目可以参考以下成熟实践，但需要改造成适合 AI 协作和领域建模的轻量版本。

### 1. Shift-left + 持续测试

DORA 的测试自动化能力强调：质量反馈不应等到“开发完成”之后，而要贯穿交付生命周期；快速、可靠的自动化测试应进入交付流水线。

对本项目的映射：

- 生成中 checklist 是一种 shift-left。
- RFC 验收样例和单测是自动化反馈。
- 复盘把后置问题转成下一次前置检查。

参考：

- https://dora.dev/capabilities/test-automation/
- https://dora.dev/capabilities/continuous-integration/

### 2. 小批量、短反馈循环

DORA 的持续集成实践强调小批量、频繁集成、自动测试和快速反馈。GitHub Flow 也鼓励在分支上小步提交，通过 PR 获取反馈。

对本项目的映射：

- RFC 不应一次性塞入过多未来范围。
- P0 应保持小而可验证。
- 实现要小步走，并尽快跑新测试和相关回归。

参考：

- https://dora.dev/capabilities/continuous-integration/
- https://docs.github.com/en/get-started/using-github/github-flow

### 3. 代码评审关注设计、复杂度和测试

Google Engineering Practices 的 code review 资料强调代码评审是一套完整文档，包含设计、实现、复杂度、测试等维度。

对本项目的映射：

- RFC 评审不只看文档完整性，也要看测试样例是否贴近框架。
- 实现评审不只看能否通过测试，也要看是否过度新增 DTO/helper/字段。
- 生成后复盘可以承担一部分“设计评审 + 过程评审”的职责。

参考：

- https://google.github.io/eng-practices/review/reviewer/

### 4. Retrospective 持续改进

Atlassian 的 retrospective playbook 强调：回顾的价值在于定期审视哪里做得好、哪里不好，并确定下一步改进动作。

对本项目的映射：

- `rfc-driven-retrospective` 不只是总结，而是把返工原因转成可执行改进。
- 每次复杂 RFC 后都可以轻量复盘一次。
- 复盘输出应落到 skill、文档或 checklist，而不是停留在口头经验。

参考：

- https://www.atlassian.com/team-playbook/plays/retrospective

## 后续讨论重点

下一步最值得改的不是生产代码，而是两个 skill 的“前置侦察 + 复用优先 + 测试惯例”部分。

建议优先讨论：

1. 哪些规则应该写进 `interactive-rfc-generation`。
2. 哪些规则应该写进 `rfc-driven-development`。
3. 哪些规则应该单独做成项目知识文档。
4. 哪些规则属于用户偏好，需要作为长期协作约定保留。

核心方向：

- 少发明接口，多发现接口。
- 少写抽象样例，多写项目真实样例。
- 少让用户重复填字段，多从上下文推导。
- 少在实现阶段才发现风格偏差，多在 RFC 阶段暴露问题。
- 把用户偏好变成 skill 的硬性检查项，而不是靠每次人工纠偏。
