## 1. 业务问题

### 1.1 配置引擎的核心职责

JMix Config Engine 的核心职责是**产品配置约束求解**。在电信设备领域（如路由器、服务器、防火墙等高度定制化产品）中，设备由大量功能模块和组件组成，每个模块有参数，模块之间存在复杂的业务规则约束。当用户给出一组输入条件后，引擎需要：

1. **补全配置**：根据业务规则自动推断用户未指定的参数和组件选择。
2. **验证合法性**：确保最终配置方案满足所有业务规则。
3. **优化配置**：在多个可行方案中，根据优先级规则选出最优方案。

然而，在实际业务中，这两个需求都会遇到挑战。

### 1.2 业务问题一：部分满足

**问题描述**：用户的输入条件与业务规则之间可能存在矛盾，导致**没有任何方案能同时满足所有条件**。

在电信产品配置场景中，这是一个极其常见的问题。以企业级路由器选配为例：

**场景一：企业路由器配置**

```
业务规则（Module Catalog）:
  - rule1: 如果选用转发性能 > 200Gbps 的路由器，则必须搭配散热模块（风扇数 >= 4）
  - rule2: 最大支持风扇数为 3（物理规格限制）
  - rule3: 转发性能 > 200Gbps 的型号需要 10GE 光模块

用户输入:
  - 转发性能需求: 250Gbps (>200Gbps)
  - 风扇数限制: 3 个（机房环境约束）

分析:
  rule1 + 用户输入 → 风扇数 >= 4
  rule2 → 风扇数 <= 3
  3 >= 4 ? → 无解！
```

此时，如果系统仅返回"无解"，用户将面临**完全无法理解为何配置失败**的困境。用户并不知道是哪个规则导致了冲突，也不知道如何调整输入。

**期望的系统行为**：即使所有规则无法同时满足，系统也应该能够找到一个"尽可能满足大部分规则"的配置方案，并告诉用户：**哪些规则被放宽了，以及在放宽这些规则后，最优配置是什么**。

例如，系统可以告诉用户："如果将风扇数要求放宽到 4 个（超过物理限制），则 250Gbps 的路由器可以正常工作。当前推荐配置：250Gbps 路由器 + 4 风扇散热模块 + 10GE 光模块。"

这就是**部分满足（Partial Satisfaction）**的核心价值——系统不再给出"全有或全无"的二值判断，而是给出一个有信息量的中间状态，帮助用户做出业务决策。

### 1.3 业务问题二：规则冲突检查与定位

**问题描述**：在复杂产品中，规则数量可能达到数百甚至上千条。当配置失败时，定位**哪条规则与哪条规则之间存在冲突**是一个极其困难的问题。

**场景二：数据中心交换机配置**

考虑一个更复杂的场景，涉及数十个功能模块和数百条规则：

```
系统包含以下模块分类:
  - 交换引擎 (switching_engine): 多个型号，转发能力各异
  - 端口模块 (port_module): 多种光模块类型，速率各异
  - 电源模块 (power_supply): 多种功率，冗余配置各异
  - 散热模块 (cooling_module): 多种散热方式，功耗管理各异
  - 管理平面 (mgmt_plane): 多个品牌，管理协议各异
  - 安全模块 (security_module): 多种防火墙规格，吞吐各异
  - ...

业务规则:
  rule_port_01: 端口模块总带宽 > 400Gbps → 交换引擎转发能力 >= 600Gbps
  rule_port_02: 端口模块总带宽 > 400Gbps → 电源模块功率 >= 1500W
  rule_port_03: 10GE 光模块数量 > 8 → 端口模块必须支持 QSFP+
  rule_power_01: 电源模块功率 >= 1500W → 散热模块功耗处理能力 >= 1600W
  rule_cool_01: 散热模块功耗处理 >= 1600W → 机箱深度 >= 500mm
  rule_mgmt_01: 管理平面品牌 = "H3C" → 仅支持 H3C 私有管理协议
  rule_sec_01: 安全模块吞吐 >= 100Gbps → 交换引擎支持硬件 ACL
  rule_sec_02: 安全模块吞吐 >= 100Gbps → 电源模块必须冗余配置
  ...

用户输入:
  - 端口模块: 总带宽 480Gbps（>400Gbps）
  - 管理平面: 品牌 "H3C"
  - 安全模块: 吞吐 80Gbps（<100Gbps，低于规则要求）
```

在这个例子中，用户输入与多条规则产生了矛盾：安全模块吞吐不足、管理协议受限……但更关键的是，**这些输入之间本身也存在传递依赖**：安全模块 80Gbps 与"吞吐 >= 100Gbps → 硬件 ACL → 交换引擎要求"的传递依赖链条。

**期望的系统行为**：系统应该能够自动追踪这些规则之间的依赖关系，并精确指出：

```
冲突诊断报告:
  - 冲突规则 1: rule_sec_01（安全模块吞吐 >= 100Gbps → 交换引擎支持硬件 ACL）
    来源: 规则表达式, 权重: MEDIUM
    自然语言: "大吞吐安全模块需要硬件 ACL 支持"

  - 冲突规则 2: rule_sec_02（安全模块吞吐 >= 100Gbps → 电源模块必须冗余配置）
    来源: 规则表达式, 权重: MEDIUM
    自然语言: "大吞吐安全模块需要冗余电源"

  - 冲突输入: 用户选择了 80Gbps 安全模块
    来源: 用户输入约束, 权重: SMALL

  部分满足方案（在放宽"安全模块吞吐 >= 100Gbps"后）:
    交换引擎: SE-600G (600Gbps 转发能力)
    端口模块: 480Gbps 总带宽
    电源模块: 1500W
    散热模块: 1600W 功耗处理
    管理平面: H3C 私有协议
    安全模块: 80Gbps（非冗余电源配置）
```

### 1.4 业务问题总结

| 业务问题      | 核心挑战                           | 期望行为                    |
| --------- | ------------------------------ | ----------------------- |
| **部分满足**  | 用户输入与业务规则矛盾，无法得到完整解            | 找到"最小冲突"的可行部分解          |
| **冲突定位**  | 数百条规则交织，难以人工定位冲突根源             | 精确指出哪些规则被违反，并给出违反的原因链   |
| **部分解展示** | 简单的"无解"对用户无参考价值                | 提供放宽规则后的候选方案，帮助用户调整输入   |
| **多实例场景** | 同一模块分类的多个实例可能有不同的过滤条件，部分成功部分失败 | 部分分类正常求解，失败分类携带错误信息继续输出 |

## 2. 约束求解的解决方案

### 2.1 从业务规则到数学模型

约束求解的核心思想是：将产品配置问题建模为一个**约束满足与优化问题（Constraint Satisfaction and Optimization Problem, CSOP）**，然后交给专门的数学求解器来计算。

**变量定义**：

- 对于每个组件 `Part`，定义整数变量 `qty_i ∈ [0, maxQty_i]` 表示数量
- 对于每个枚举参数 `Para`，定义整数变量 `value_j ∈ optionIds_j` 表示选中值
- 对于每个布尔条件（如"是否选中某选项"），定义布尔变量 `b_k ∈ {0, 1}`

**约束定义**：每条业务规则被翻译为一个或多个数学约束：

```
rule1: x.qty + y.qty < 20   →   x_qty + y_qty ≤ 19
rule2: x.qty - y.qty > 10   →   x_qty - y_qty ≥ 11
rule3: y.qty > 45            →   y_qty ≥ 46
```

**求解器任务**：找到一个变量赋值 `(x_qty, y_qty, ...)`，使得所有约束同时满足。

### 2.2 部分满足的求解策略

当原始模型无解时，部分满足通过**松弛变量（Relaxation Variables）**实现：

```
原始约束: x_qty + y_qty ≤ 19
          ↓ 添加松弛变量 r1 ∈ {0, 1}
松弛约束: x_qty + y_qty ≤ 19 + M * r1

其中 M 是一个很大的常数。当 r1=0 时，约束正常执行；
当 r1=1 时，约束被放松（rhs变成一个极大的数）。
```

目标函数变为：

```
minimize Σ(r_i * weight_i)
```

即**最小化被放松约束的加权和**。这样求解器会优先放松权重低的约束，保留权重高的核心业务规则。

### 2.3 冲突诊断的求解策略

冲突诊断基于松弛变量的取值：

1. 原始模型无解（`CpSolverStatus.INFEASIBLE`）
2. 启动松弛诊断流程：给所有可诊断约束附加松弛变量
3. 求解松弛模型（目标：最小化被放松约束）
4. 松弛变量值为 `true` 的约束 → 被识别为**冲突约束**
5. 从冲突约束集合构建诊断报告，并输出部分可行解

## 3. 约束求解核心原理：通俗图解

> 本章用日常生活的通俗例子，解释约束求解和冲突诊断的核心原理。不熟悉电信产品配置或约束规划的读者，可以从这里开始建立直观理解。

### 3.1 什么是约束？——从杯子说起

想象你面前有一张桌子，桌上有**三个杯子**（记作 A、B、C），每个杯子可以放 **0 到 5 颗糖**。

我们给这张桌子设定几条"规则"：

```
规则1: A 和 B 的糖加起来至少 8 颗       →  A + B ≥ 8
规则2: C 的糖不超过 3 颗                  →  C ≤ 3
规则3: 如果 A 的糖超过 4 颗，那么 C 至少 2 颗   →  A > 4 → C ≥ 2
规则4: 三种杯子平均每杯不超过 4 颗糖      →  (A + B + C) / 3 ≤ 4  → A + B + C ≤ 12
规则5: 如果 B 的糖超过 3 颗，A 必须少于 5 颗   →  B > 3 → A ≤ 4
```

现在我们问求解器：**有没有一种分配方案同时满足所有 5 条规则？**

这就是一个典型的**约束满足问题（CSP）**。求解器会搜索所有可能的 `(A, B, C)` 组合，找到满足规则的解。

如果我们再加一条规则：

```
规则6: A + B ≥ 10
```

此时 6 条规则同时满足就变得困难了——需要同时满足 `A + B ≥ 8` 和 `A + B ≥ 10`，即 `A + B ≥ 10`，同时还要满足 `A + B ≤ 12`（从规则4推导）。求解器可能还能找到解。

但如果我们再加一条规则：

```
规则7: A + B ≤ 9
```

此时规则6和规则7直接矛盾：`A + B ≥ 10` 且 `A + B ≤ 9` —— 这在数学上不可能。此时求解器会返回**无解**。

问题来了：**是哪条规则和哪条规则冲突了？** 是规则6和规则7直接矛盾，还是规则6和规则4矛盾，还是多条规则组合导致的？人工排查非常困难——而这正是配置引擎需要解决的问题。

### 3.2 从 5 条规则开始：逐步引入冲突

为了更清楚地理解冲突是如何产生的，我们把问题简化，用一个更小的例子：

**初始状态**：只有前 5 条规则（A、B、C 三个杯子）

```
规则1: A + B ≥ 8
规则2: C ≤ 3
规则3: A > 4 → C ≥ 2
规则4: A + B + C ≤ 12
规则5: B > 3 → A ≤ 4
```

求解器可以找到一个解，例如：

```
A = 5,  B = 4,  C = 2

验证:
  规则1: 5 + 4 = 9 ≥ 8 ✓
  规则2: 2 ≤ 3 ✓
  规则3: A=5 > 4, 所以 C ≥ 2, C=2 ✓
  规则4: 5 + 4 + 2 = 11 ≤ 12 ✓
  规则5: B=4 > 3, 所以 A ≤ 4, 但 A=5 → ✗ 冲突！
```

等等，规则5也被违反了。调整一下：

```
A = 4,  B = 4,  C = 2

验证:
  规则1: 4 + 4 = 8 ≥ 8 ✓
  规则2: 2 ≤ 3 ✓
  规则3: A=4 不大于 4, 跳过 C ≥ 2 的要求 ✓
  规则4: 4 + 4 + 2 = 10 ≤ 12 ✓
  规则5: B=4 > 3, 所以 A ≤ 4, A=4 ✓

全部通过！解: (A=4, B=4, C=2)
```

**这是正常求解流程**——求解器搜索变量空间，找到一个满足所有规则的解。

**现在引入用户输入**：用户说"我想在 A 中放 6 颗糖"。

用户输入被转化为一个约束：`A = 6`。加上原来的 5 条规则：

```
用户输入: A = 6
规则1:  A + B ≥ 8    →  B ≥ 2
规则2:  C ≤ 3
规则3:  A > 4 → C ≥ 2  →  A=6 > 4, 所以 C ≥ 2
规则4:  A + B + C ≤ 12  →  6 + B + C ≤ 12  →  B + C ≤ 6
规则5:  B > 3 → A ≤ 4    →  A=6, 所以 B > 3 → A ≤ 4 矛盾
```

这里有两种解读：

**解读 A（如果 A=6 来自用户硬性输入）**：

- 规则5说"如果 B > 3，那么 A 必须 ≤ 4"，但 A=6 > 4
- 所以如果 B > 3，规则5就矛盾了
- 要满足规则5，必须 B ≤ 3

**解读 B（如果 A=6 来自用户硬性输入，且无法修改）**：

- 规则5无法满足 → 无解

假设用户坚持 A=6 不变，那么唯一可能的解是 B ≤ 3。但再看规则1要求 B ≥ 2，所以 B 只能是 2 或 3。取 B=2 试试：

```
A=6, B=2, C=?

规则1: 6 + 2 = 8 ≥ 8 ✓
规则4: 6 + 2 + C ≤ 12 → C ≤ 4
规则2: C ≤ 3
规则3: A > 4 → C ≥ 2

所以 C 的可行范围是: max(0, 2) ≤ C ≤ min(3, 4) → C ∈ {2, 3}
```

等等，这说明在用户输入 A=6 的情况下，**仍然有解**：`(A=6, B=2, C=2)` 或 `(A=6, B=3, C=2)`。

但如果我们把规则4改成更严格的约束：

```
规则4': A + B + C ≤ 10（改为 ≤ 10）
```

用户输入仍为 `A = 6`：

```
规则1: A + B ≥ 8  →  B ≥ 2
规则4': 6 + B + C ≤ 10  →  B + C ≤ 4
规则3: A > 4 → C ≥ 2
规则2: C ≤ 3

综合:
  B ≥ 2
  B + C ≤ 4
  C ≥ 2
  C ≤ 3

如果 C=2 → B ≤ 2 → B ≥ 2 → B=2 ✓
如果 C=3 → B ≤ 1 → B ≥ 2 → 无解

所以 C 只能是 2，B 只能是 2，解为 (A=6, B=2, C=2)。
```

仍然有解。继续加大难度：

```
规则4'': A + B + C ≤ 8（改为 ≤ 8）

此时:
  6 + B + C ≤ 8  →  B + C ≤ 2
  规则1: B ≥ 2
  规则3: C ≥ 2

B + C ≤ 2 且 B ≥ 2 且 C ≥ 2
→ B ≥ 2, C ≥ 2 → B + C ≥ 4 > 2 → 无解！
```

**此时系统返回无解**。但用户会困惑："为什么无解？是哪里出问题了？"

### 3.3 松弛变量：给每条规则装一个"开关"

这就是**松弛变量（Relaxation Variable）**的核心思想——给每条规则安装一个"开关"，允许在紧急情况下"暂时绕过"该规则。

**给规则装开关的过程如下**：

```
原始规则: A + B + C ≤ 8

添加松弛变量 r4'' ∈ {0, 1}，引入一个大数 M = 100：

松弛后规则: A + B + C ≤ 8 + M * r4''

  当 r4'' = 0 时: 约束变为 A + B + C ≤ 8（规则正常执行）
  当 r4'' = 1 时: 约束变为 A + B + C ≤ 108（规则被"绕过"）
```

我们将**所有 5 条规则**都装上松弛变量：

```
规则1':  A + B ≥ 8  - M * r1    →  A + B ≥ 8 - M*r1   (r1=1时无下界)
规则2':  C ≤ 3  + M * r2         →  C ≤ 3 + M*r2
规则3':  (A > 4 → C ≥ 2)          →  用蕴含变量实现 + 松弛
规则4'': A + B + C ≤ 8 + M * r4'' →  (r4''=1时无上界)
规则5:   (B > 3 → A ≤ 4)          →  用蕴含变量实现 + 松弛
用户输入: A = 6
```

**关键变化**：约束的右端变成了一个表达式，而不是固定常数。当松弛变量 = 1 时，约束的"强度"被大幅削弱，几乎相当于没有约束。

**然后我们给求解器一个新的目标**：最小化被激活的松弛变量数量。

```
目标函数: minimize (r1*1 + r2*1 + r3*1 + r4''*1 + r5*1)
```

即求解器需要**在满足所有（松弛后的）约束的前提下，让松弛变量尽可能少地取 1**。

### 3.4 松弛之后：求解器看到了什么

现在求解器面对的是以下模型：

```
变量:
  A = 6（用户输入，固定）
  B ∈ [0, 5]
  C ∈ [0, 5]
  r1, r2, r3, r4'', r5 ∈ {0, 1}

约束:
  A + B ≥ 8 - 100*r1
  C ≤ 3 + 100*r2
  A + B + C ≤ 8 + 100*r4''
  (B > 3 → A ≤ 4) + 松弛
  (A > 4 → C ≥ 2) + 松弛

目标: minimize (r1 + r2 + r3 + r4'' + r5)
```

求解器开始求解：

**第一步**：尝试所有 r = 0（不放松任何规则）

```
如果 r1=r2=r3=r4''=r5=0：
  A + B ≥ 8 → B ≥ 2
  C ≤ 3
  A + B + C ≤ 8 → 6 + B + C ≤ 8 → B + C ≤ 2

已知 A=6, B ≥ 2, C ≥ 2（规则3的前件满足），但 B ≥ 2 且 C ≥ 2 → B + C ≥ 4 > 2，矛盾！

所以 r = 0 不可行。
```

**第二步**：求解器开始尝试让某些 r = 1

由于目标是最小化 Σr，求解器会先尝试只放松权重最低的规则。

我们假设权重相同（都 = 1），则求解器会找到一个**使 Σr 最小**的组合。

尝试 r4'' = 1（放松规则4''），其余为 0：

```
如果 r4''=1, r1=r2=r3=r5=0：
  A + B ≥ 8 → B ≥ 2
  C ≤ 3
  A + B + C ≤ 108（相当于无上界）
  A > 4 → C ≥ 2（从 A=6 推导 C ≥ 2）
  B > 3 → A ≤ 4（从 A=6 推导 B ≤ 3）

综合:
  B ≥ 2
  B ≤ 3   → B ∈ {2, 3}
  C ≥ 2
  C ≤ 3   → C ∈ {2, 3}

有解！例如 (A=6, B=2, C=2)，Σr = 1
```

尝试只放松另一条规则（如 r1=1）：

```
如果 r1=1, r4''=0, 其余为0：
  A + B ≥ -92（相当于无下界）
  C ≤ 3
  A + B + C ≤ 8 → B + C ≤ 2
  A > 4 → C ≥ 2

已知 C ≥ 2 且 C ≤ 3 → C ∈ {2, 3}
B + C ≤ 2 → 如果 C=2 → B ≤ 0, 但规则1要求 B ≥ 2 已被放松
  如果 C=3 → B ≤ -1, 不可能

所以 C 只能是 2，B 只能是 0。

解 (A=6, B=0, C=2)，Σr = 1
```

两个解的 Σr 都 = 1，但**解的内容不同**：

- 放松 r4''（规则4''）→ 解 `(A=6, B=2, C=2)`：平均糖数约束被绕过，三杯总量为 10 超过限制
- 放松 r1（规则1）→ 解 `(A=6, B=0, C=2)`：糖数和约束被绕过，总量 8 满足但 A+B=6 < 8

两个解的松弛数量相同。求解器可以选择任意一个——具体选择取决于求解器的内部启发式和搜索顺序。

### 3.5 冲突诊断：从松弛结果读出哪些规则被"绕过"了

**回到用户最关心的问题**：为什么无解？哪些规则导致了冲突？

通过松弛模型的求解结果，我们可以直接读出答案：

```
松弛模型求解结果:
  r4'' = 1  ← 规则4''被放松
  r1 = r2 = r3 = r5 = 0  ← 其他规则全部满足

部分解: (A=6, B=2, C=2)

冲突诊断报告:
  - 冲突规则: 规则4'' (A + B + C ≤ 8)
    原因: 该规则与用户输入 A=6 不兼容
    放宽说明: 允许总量达到 10（而不是上限 8）

  部分满足方案:
    A = 6 (用户输入)
    B = 2 (规则1 推导: A+B ≥ 8)
    C = 2 (规则3 推导: A>4 → C≥2)
    总糖数 = 10（超过规则4'' 的上限 8）
```

用户看到这份报告后，可以做出明智的决策：

- "我知道规则4（总量不超过8）被放宽了，放心使用这个配置"
- "我想严格遵守规则4，那需要把 A 降到 4 或 5"

**这就是松弛变量机制的核心价值**：将一个"无解/有解"的二值问题，转化为一组"哪些规则被绕过 + 绕过后的可行解"，给用户一个有价值的决策参考。

### 3.6 小结

| 步骤            | 做了什么                     | 结果                                |
| ------------- | ------------------------ | --------------------------------- |
| **1. 建模**     | 将业务规则翻译为数学约束             | `A + B + C ≤ 8`, `A + B ≥ 8`, ... |
| **2. 求解**     | 求解器搜索满足所有约束的变量组合         | 无解（A=6 时规则1+3+4'' 矛盾）             |
| **3. 添加松弛变量** | 给每条规则加一个开关 `r_i ∈ {0,1}` | `A + B + C ≤ 8 + M*r4''`          |
| **4. 重新求解**   | 最小化 Σ(r_i * weight_i)    | Σr = 1，找到部分解 (A=6, B=2, C=2)      |
| **5. 诊断**     | 读取 `r_i = 1` 的规则         | 规则4'' 被放宽                         |
| **6. 展示**     | 报告冲突规则 + 部分解             | 告诉用户"放宽总量上限后可以这样配置"               |

## 4. 约束生成原理（概要设计）

### 4.1 数据模型：业务域到配置域的映射

JMix Config Engine 将产品配置建模为**四层嵌套结构**：

```
Module（模块）
 ├── Para[]           ← 模块级参数（枚举/整数等）
 ├── Part[]           ← 原子组件（价格、数量约束）
 ├── PartCategory[]   ← 组件分类（可递归嵌套）
 │    ├── Para[]
 │    ├── Part[]
 │    └── PartCategory[]   ← 子分类（可递归）
 └── Rule[]           ← 业务规则
```

**继承体系**：

```
ProgrammableObject(code, fatherCode, defaultValue, description, sortNo)
 └─ Onto
      ├── Para（参数）
      ├── Part（组件）
      └── ModuleBase
           ├── Module（模块）
           └── PartCategory（组件分类）
```

`Onto` 是核心数据容器，管理 `Para[]`、`Rule[]` 和 `DynamicAttribute[]` 的关联。`ModuleBase` 扩展 `Onto`，增加了 `Part[]` 和 `PartCategory[]` 的层级管理。值得注意的是，**`PartCategory` 本身也实现了 `IModule` 接口**，因此组件分类可以递归包含子分类，形成任意深度的层级结构。

**配置实例模型**：

```
ModuleInst（模块实例）
 ├── ParaInst[]           ← 参数实例值
 ├── PartInst[]           ← 组件实例（数量、选中状态）
 ├── PartCategoryInst[]   ← 组件分类实例（可递归）
 └── PriorityAttrValue[]  ← 优先级属性值
```

### 4.2 变量系统：核心变量

对于产品配置中的每一个 `Para` 和 `Part`，引擎会自动创建一组 CP-SAT 变量。

**Para 变量**（针对枚举型参数）：

| 变量名             | 类型        | 含义                  |
| --------------- | --------- | ------------------- |
| `para_value`    | `IntVar`  | 参数的实际值（如选中哪个选项的 ID） |
| `para_opt_{id}` | `BoolVar` | 每个选项是否被选中           |

枚举参数通过 `para_value` 变量和一组 `para_opt_*` 选项变量联动：

```
para_value == option_id   ⟺   para_opt_{id} == 1
para_value != option_id   ⟺   para_opt_{id} == 0
```

**Part 变量**（针对每个组件）：

| 变量名                  | 类型        | 含义                         |
| -------------------- | --------- | -------------------------- |
| `pt_qty_{code}`      | `IntVar`  | 组件数量，取值 `[0, maxQuantity]` |
| `pt_selected_{code}` | `BoolVar` | 组件是否被选中                    |

组件的数量和选中状态通过约束联动：

```
pt_selected == 1   →   pt_qty >= 1      （选中则数量至少为1）
pt_selected == 0   →   pt_qty == 0      （未选中则数量为0）
```

**PartCategory 变量**：每个组件分类内部管理自己的 Para 和 Part 变量集合，并负责汇总计算（如 `sum4Selected`、`sum4Quantity`）。

### 4.3 约束构建：从规则 Schema 到 CP-SAT 约束

每条业务规则通过其对应的 `RuleSchema` 子类被翻译为 CP-SAT 约束。以下是几种主要规则类型的约束映射：

#### 4.3.1 兼容性规则（CompatRuleSchema）

兼容性规则有三种操作符，对应三种 CP-SAT 约束模式：

**不兼容（Incompatible）**：选 A 则不能选 B

```
规则: Incompatible(A, B)
CP-SAT:
  addImplication(A_cond, !B_cond)   // 选 A → 不选 B
  addImplication(B_cond, !A_cond)   // 选 B → 不选 A
```

**依赖（Requires）**：选 A 则必须选 B

```
规则: Requires(A, B)
CP-SAT:
  addImplication(A_cond, B_cond)   // 选 A → 选 B
```

**共存（CoDependent）**：选 A 则必须选 B，且选 B 则必须选 A

```
规则: CoDependent(A, B)
CP-SAT:
  addImplication(A_cond, B_cond)   // 选 A → 选 B
  addImplication(B_cond, A_cond)   // 选 B → 选 A
```

#### 4.3.2 线性不等式规则（CodeRuleSchema）

业务规则中的线性数量约束（如 `x.qty + y.qty < 20`）通过 `AlgCPLinearExpr` 构建线性表达式：

```java
AlgCPLinearExpr sumExpr = model.newLinearExpr("sum_x_y");
sumExpr.addTerm(x_qty, 1);    // 系数 1
sumExpr.addTerm(y_qty, 1);    // 系数 1
model.addLessThan(sumExpr, 20);  // x + y < 20
```

#### 4.3.3 条件规则（If-Then）

带条件的前件-后件约束（`if condition then consequent`）通过 `onlyEnforceIf` 实现：

```java
// if x.qty > 10 then y.qty > 7
BoolVar xGt10 = model.newBoolVar("x_gt_10");
model.addGreaterThan(x_qty, 10).onlyEnforceIf(xGt10);
model.addLessOrEqual(x_qty, 10).onlyEnforceIf(xGt10.not());
model.addGreaterThan(y_qty, 7).onlyEnforceIf(xGt10);
```

当 `xGt10 = false` 时（即 `x_qty <= 10`），后件约束被跳过；当 `xGt10 = true` 时，后件约束强制执行。

#### 4.3.4 求和约束（sumFunConstraint）

组件分类的属性汇总约束（如"所有端口模块总带宽 >= 400Gbps"）通过遍历组件变量并加权求和：

```java
// sum(Bandwidth_i * qty_i for i in port_modules) >= 400
PartAlgCPLinearExpr sumExpr = model.newPartLinearExpr("sum_bandwidth");
for (PartVarImpl partVar : partVars) {
    int bandwidth = partVar.getAttr4Int("Bandwidth");  // 属性系数
    sumExpr.addTerm(categoryCode, partVar, partVar.qty, bandwidth, "sum_bandwidth");
}
model.addGreaterOrEqual(sumExpr, 400);
```

### 4.4 目标函数：优先级与松弛

当存在多个可行解时，引擎通过**优先级规则（PriorityRuleSchema）**选择最优解。

优先级规则的目标函数由 `PartAlgCPLinearExpr` 构建，支持两种聚合方式：

- `sum4Selected(attrCode, filter)`：按属性加权统计选中的组件数量
- `sum4Quantity(attrCode, filter)`：按属性加权统计组件总量

```
目标: maximize Σ(attr_i * qty_i)     →  MAX 策略，乘以 -1 变为 minimize
目标: minimize Σ(attr_i * qty_i)     →  MIN 策略，直接 minimize
```

**松弛目标函数**：在冲突诊断模式下，所有松弛变量的加权和被设为独立的目标函数，覆盖原优先级目标：

```java
// 松弛目标函数：minimize Σ(relaxVar_i * weight_i)
AlgCPLinearExpr relaxExpr = model.newLinearExpr("relax_objective");
for (RelaxVar rv : relaxVarMap.values()) {
    relaxExpr.addTerm(rv.getValue(), rv.getWeight());
}
model.minimize(relaxExpr);
```

**松弛变量权重等级**：

| 权重常量            | 值    | 用途                 |
| --------------- | ---- | ------------------ |
| `WEIGHT_SMALL`  | 1    | 系统级约束（输入条件）        |
| `WEIGHT_MEDIUM` | 25   | 自定义业务规则            |
| `WEIGHT_BIG`    | 1000 | 高优先级规则             |
| `WEIGHT_ADDER`  | 1000 | 冲突叠加（鼓励多次诊断发现更多冲突） |

权重越大，求解器越不愿意放松该规则。`WEIGHT_ADDER` 在二次松弛诊断中使用，对已被识别为冲突的规则进一步增加权重，促使求解器寻找替代冲突集。

### 4.5 实例：完整约束生成流程

以下通过一个具体例子，演示从业务规则到 CP-SAT 模型的完整生成过程。

**业务场景：企业级路由器配置**

```
Module: 企业路由器配置
  Para:
    - forwarding_perf: 枚举 {100Gbps, 200Gbps, 400Gbps}
    - fan_count: 枚举 {2, 3, 4}

  Part:
    - mgmt_plane_h3c: 管理平面（H3C品牌），支持私有协议
    - mgmt_plane_huawei: 管理平面（华为品牌），支持私有协议
    - port_10ge: 10GE 光模块，支持 8 个端口
    - port_40ge: 40GE 光模块，支持 4 个端口

  Rules:
    - rule1: forwarding_perf >= 400Gbps → port_40ge 被选中（需要高速端口）
    - rule2: fan_count <= 3 → port_10ge 被选中（受限散热）
    - rule3: port_10ge + port_40ge 总端口带宽 > 300Gbps
    - rule4: fan_count = 4 → 必须搭配 40GE 光模块
```

**Step 1: 变量创建**

| 变量名                     | 类型      | 域               |
| ----------------------- | ------- | --------------- |
| `P_forwarding_perf`     | IntVar  | {100, 200, 400} |
| `P_fan_count`           | IntVar  | {2, 3, 4}       |
| `PT_port_10ge_qty`      | IntVar  | [0, 1]          |
| `PT_port_10ge_selected` | BoolVar | {0, 1}          |
| `PT_port_40ge_qty`      | IntVar  | [0, 1]          |
| `PT_port_40ge_selected` | BoolVar | {0, 1}          |

**Step 2: 联动约束**

```
PT_port_10ge_selected == 1  →  PT_port_10ge_qty >= 1
PT_port_10ge_selected == 0  →  PT_port_10ge_qty == 0
```

**Step 3: 规则约束**

```
rule1 (Requires):
  forwarding_perf >= 400Gbps → port_40ge_selected == 1

  展开为:
  BoolVar fp_ge_400 = (P_forwarding_perf >= 400)
  addImplication(fp_ge_400, PT_port_40ge_selected == 1)

rule3 (线性求和):
  port_10ge_qty + port_40ge_qty > 1

  展开为:
  addGreaterThan(PT_port_10ge_qty + PT_port_40ge_qty, 1)
```

**Step 4: 求解**

```
输入: forwarding_perf = 400Gbps, fan_count = 3

求解结果:
  forwarding_perf = 400
  fan_count = 3
  port_10ge_selected = 1   ← rule2 推导（fan_count=3 <= 3）
  port_40ge_selected = 1   ← rule1 推导（fp=400 >= 400）
```

## 5. OR-Tools 搜索算法原理

### 5.1 CP-SAT 整体架构

Google OR-Tools 的 CP-SAT 求解器是一个**混合 Portfolio 求解器**，其核心融合了三大技术：

1. **约束编程（CP）**：直接处理整数域变量和复杂逻辑约束。
2. **SAT 求解（CDCL）**：利用冲突驱动子句学习（Conflict-Driven Clause Learning）进行高效搜索。
3. **线性规划（LP）**：通过 LP 松弛提供可行域边界、引导搜索方向和切平面加速。

CP-SAT 的核心设计理念来自 Peter Stuckey 的论文 **"Search is Dead: Long Live Proof"**，核心观点是：**与其在搜索树中盲目回溯，不如用 SAT 求解器的强大推理能力来精确传播约束、记录冲突**。

```
CP-SAT 求解流程：

┌──────────────────────────────────────────────────────────────────┐
│  1. 预处理（Presolve）                                            │
│     - 约束简化、域收缩、变量等价检测                               │
│     - 探测性赋值：固定变量看能传播出什么                           │
│     - 生成内部求解模型 + 通道模型（channeling model）             │
└────────────────────────────┬─────────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│  2. 变量编码（Boolean Encoding）                                  │
│     - 每个整数变量 x 被编码为两个布尔变量序列：                     │
│       - B_x=v  表示 x == v                                        │
│       - B_x≤v  表示 x <= v                                       │
│     - 整数变量的约束 ⟺ 布尔变量的合取范式（CNF）                  │
└────────────────────────────┬─────────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│  3. 搜索循环（CDCL + 约束传播）                                    │
│     - 分支决策（Branch）：选择一个变量和一个值                      │
│     - 约束传播（Propagate）：应用所有约束，消除不一致值             │
│     - 冲突检测（Conflict）：检测到域冲突时，记录冲突子句并回跳       │
│     - 重复直到找到解或证明无解                                      │
└────────────────────────────┬─────────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│  4. LP 松弛集成（切平面 + 启发式）                                 │
│     - LP 松弛作为约束传播器（低优先级）                            │
│     - LP 不可行射线 → 切平面约束                                   │
│     - LP 影子价格（reduced cost）→ 分支启发式                     │
└────────────────────────────┬─────────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│  5. Portfolio 并行搜索                                            │
│     - 多个 worker 运行不同的搜索策略                               │
│     - 共享边界、可行解和冲突子句                                   │
│     - LNS（Large Neighborhood Search）用于深耕                     │
└──────────────────────────────────────────────────────────────────┘
```

### 5.2 Presolve 预处理阶段

CP-SAT 在正式搜索之前执行 Presolve 预处理，其目标是在**不改变问题解集**的前提下，简化问题规模。Presolve 包含以下关键步骤：

**步骤 1：约束展开与分解**

高级约束（如 `element`、`table`、`automaton`、`inverse`）被展开为基础的线性约束集合。这类似于 Minizinc 到 FlatZinc 的编译过程。

**步骤 2：域收缩（Domain Reduction）**

利用约束的内在性质直接缩小变量的取值范围。例如，约束 `x + y >= 10` 且已知 `x ∈ [0, 5]`，则可以推出 `y >= 5`。

**步骤 3：变量等价与仿射关系检测**

如果两个变量的取值始终相同（如 `x == y` 在所有约束中始终一起出现），Presolve 会将其合并为一个变量。如果存在仿射关系（如 `x == 2*y + 1`），也会被识别并简化。

**步骤 4：探测性赋值（Probing）**

临时固定某个变量的值，模拟传播过程。如果传播导致矛盾，则说明该变量值不能取——可以将该值从域中永久移除。

**步骤 5：生成双模型**

Presolve 输出两个模型：

- **内部求解模型**：简化后的紧凑模型，用于实际求解。
- **通道模型（Channeling Model）**：内部变量与原始变量之间的映射，用于将求解结果映射回原始问题。

### 5.3 整数变量的布尔编码

CP-SAT 将整数变量通过**双序列编码（Two-Phase Encoding）**转换为布尔变量。每个整数变量 `x ∈ [l, u]` 被编码为两组布尔变量：

**第一组：域值变量（Domain Variables）**

```
B_{x=v}  表示  x == v
```

对于每个可能的值 `v ∈ [l, u]`，创建一个布尔变量。约束 `x == v ⟺ B_{x=v} == 1`。

**第二组：序关系变量（Order Variables）**

```
B_{x≤v}  表示  x <= v
```

对于每个边界值 `v ∈ [l, u]`，创建一个布尔变量。注意 `B_{x≤v}` 和 `B_{x≤v-1}` 之间存在单调关系：`B_{x≤v} >= B_{x≤v-1}`。

**两种编码的转换关系**：

```
x == v   ⟺   (x >= v) ∧ (x <= v)
         ⟺   B_{x≤v} ∧ ¬B_{x≤v-1}
```

**为什么需要两组？**

- 域值编码适合等式约束（`x == v`），但会导致 O(n) 个变量（n 为域大小）。
- 序关系编码适合不等式约束（`x <= v`），变量数同样为 O(n)。
- CP-SAT 通过 **_lazy addition_**（延迟添加）策略，只在需要时添加相关布尔变量，控制编码规模。

### 5.4 约束传播（Constraint Propagation）

约束传播是 CP 的核心机制，也是 CP-SAT 比纯 SAT 求解器更强大的关键所在。

**基本原理**：当一个变量的域发生变化时，通知所有相关约束；每个约束根据新信息**剪枝**其他变量的域。

**举例说明**：假设有约束 `x + y < 20`，当前域为 `x ∈ [5, 15]`、`y ∈ [8, 18]`。

1. 假设搜索决策将 `x` 固定为 `15`。
2. 传播器检测到约束 `x + y < 20`，计算 `y < 5`。
3. 将 `y` 的域从 `[8, 18]` 缩减为 `[8, 4]`——产生空域。
4. 检测到矛盾，记录冲突。

**传播器的精确性**：CP-SAT 的传播是**精确的（Exact）**，使用 64 位整数算术，避免了浮点误差。在传播过程中，系统记录每个传播决策的**解释（Explanation）**，即哪些变量赋值导致了当前域的收缩。这些解释在冲突时会被转换为布尔子句，添加到 SAT 求解器的学习中。

**传播与 SAT 的区别**：传统 SAT 求解器只能处理布尔变量和析取（OR）约束。CP-SAT 通过将 CP 传播器产生的每个域收缩解释为一个布尔蕴含（`B_{x≤v} ⇒ B_{y≤w}`），将 CP 的推理能力嫁接到 SAT 求解器的 CDCL 框架中。

### 5.5 懒子句生成（Lazy Clause Generation, LCG）

LCG 是 CP-SAT 最重要的理论创新，由 Peter Stuckey 等人提出。其核心思想是：**不要把约束的传播留给 CP 传播器独自处理，而是将传播过程中学到的逻辑关系（子句）懒添加到 SAT 求解器中**。

**传播即子句生成**：

```
约束: x + y <= 10
当前: x = 8（已固定）

传播过程:
  y <= 2（从 8 + y <= 10 推导）

生成子句:
  (x >= 8) ⇒ (y <= 2)
  即: (¬B_{x≤7}) ⇒ (B_{y≤2})
  等价于 CNF 子句: B_{x≤7} ∨ B_{y≤2}
```

这个子句 `B_{x≤7} ∨ B_{y≤2}` 被添加到 SAT 求解器的子句库中。此后，SAT 求解器的 CDCL 机制可以直接利用这个子句进行推理和冲突分析，无需再次运行 CP 传播器。

**懒添加的含义**："懒"体现在——只有当传播器实际执行了推理（产生了新知识）时，才会生成对应的子句并添加到 SAT 层。如果某个约束的当前赋值已经满足，该约束不会产生新子句。

### 5.6 CDCL SAT 求解器

CP-SAT 的底层 SAT 求解器采用 **CDCL（Conflict-Driven Clause Learning）** 算法，这是现代 SAT 求解器的工业标准。

**决策循环（Decision Loop）**：

```
1. 传播（Propagate）：
   运行所有活跃约束的传播器
   如果产生冲突 → 进入冲突分析

2. 决策（Decide）：
   如果所有变量已固定 → 找到解！
   否则 → 选择一个未赋值变量和一个赋值方向（branch）
   当前决策层级 +1

3. 重复 1-2
```

**冲突分析（Conflict Analysis）**：当传播导致矛盾（如某个变量的域变为空）时，CDCL 执行冲突分析：

```
1. 从冲突约束开始，回溯传播链
2. 找出导致冲突的决策集合
3. 计算这些决策的最小冲突子集（1UIP 规则）
4. 将冲突子集转换为学习子句（learned clause）
5. 非时序回跳（non-chronological backjump）到正确的决策层级
6. 将学习子句添加到子句库
```

**时序回跳 vs 非时序回跳**：

- **时序回跳**：从决策 1→2→3→...→15→冲突，撤回到决策 14，然后尝试另一个值。所有 1-14 的中间工作都被丢弃。
- **非时序回跳**：分析发现真正导致冲突的决策是 12（而非 15），则直接回跳到 12，保留 13-15 的部分工作。

**VSIDS 启发式**：CP-SAT 使用 VSIDS（Variable State Independent Decaying Sum）选择分支变量。该启发式根据变量在冲突子句中的出现频率更新分数，优先分支"活跃"变量。

### 5.7 LP 松弛与切平面

CP-SAT 集成了一个 **Simplex（单纯形法）** 求解器来计算当前 CP 模型的线性规划松弛。虽然 LP 松弛在 CP 问题上的界可能不如在 MIP 问题上的界紧（因为 CP 变量是离散的），但它提供了以下关键能力：

**不可行性检测**：如果 LP 松弛本身无解，则 CP 问题必然无解。LP 的对偶射线（dual ray）可以用来生成一个切平面约束，直接证明冲突。

**边界收紧**：LP 松弛可以计算变量的可行域边界（通过 reduced cost fixing）。如果 LP 解中某个变量的 reduced cost 很大，可以将该变量固定或缩小其搜索范围。

**分支引导**：LP 松弛的最优解和 reduced costs 可以作为分支启发式的输入，引导搜索朝向更有可能包含最优解的区域。

**切平面（Cuts）**：CP-SAT 在根节点应用 MIP 风格的切平面（Gomory cuts、MIR cuts、cover cuts 等）。这些切平面从 LP 松弛的不等式系统中推导出来，用于切割掉不可行区域。

切平面的生成遵循以下通用框架：

1. 将多个整数 LP 约束聚合成单个不等式（`Σ terms <= rhs`）。
2. 使用补集、隐含界和正变量重写。
3. 应用超加性函数 `f()` 切割。
4. 将切割约束转换为原始变量形式。

CP-SAT 与 MIP 求解器的关键区别在于：CP-SAT 的一切计算都使用整数算术（`int64`/`int128`），没有 epsilon，避免了浮点精度问题。

### 5.8 Portfolio 并行搜索

CP-SAT 采用 **Portfolio 架构**进行并行求解，同时运行多个不同策略的 worker：

**Worker 类型分配**：

- 前 X 个 worker：运行通用的搜索策略（不同的分支启发式、不同的变量选择顺序）。
- 剩余 worker：全部运行 **LNS（Large Neighborhood Search）**策略。

**信息共享**：多个 worker 之间共享：

- 发现的可行解（upper bound）
- 变量边界（lower/upper bound）
- 学习到的冲突子句

**搜索策略多样性**：不同 worker 采用不同的分支策略：固定顺序、随机化、基于 LP 松弛分数、基于伪成本、基于影响分析等。

### 5.9 大邻域搜索（LNS）

LNS（Large Neighborhood Search）是 CP-SAT 的另一核心组件，特别适合深耕已知的可行解邻域。

**基本思想**：给定一个可行解，随机"破坏"其中一部分变量的赋值，然后重新求解。如果快速找到了更好的解，则继续迭代；否则扩大破坏范围。

**在 CP-SAT 中的实现**：使用 **RINS（Relaxation Induced Neighborhood Search）** 技术：

1. 取当前最好的可行解（ incumbent solution）。
2. 将与 LP 松弛解一致的变量固定下来。
3. 对不一致的变量进行分支搜索。
4. 如果找到更好的解，更新 incumbent 并重复。

LNS 特别适合**难问题的深耕**：当全局搜索已经找到了不错但可能不是最优的解时，LNS 可以在该解附近进行精细化搜索。

### 6.3 算法模型层（algmodel）

`algmodel` 包是 OR-Tools CP-SAT 与业务模型之间的关键桥梁。

**AlgCPModel**：封装 OR-Tools `CpModel`，核心职责包括：

- 变量注册（`newIntVar`、`newBoolVar`）
- 约束跟踪（`constraints` 列表）
- 松弛变量管理（`relaxationVarMap`）
- 当前规则作用域（`currentRelaxVar`、`currentRelaxVarName`）

**约束松弛机制**（核心创新）：

```
model.setRelax4CustomRule("rule1")
  → currentRelaxVar = newBoolVar("relax_rule1")
  → currentRelaxVarName = "relax_rule1"
  → 注册到 relaxationVarMap{ "relax_rule1": RelaxVar(weight=MEDIUM) }

model.addLessThan(expr, 20)
  → AlgCPConstraint( cpConstraint, currentRelaxVar, "relax_rule1" )
  → cpConstraint.onlyEnforceIf(currentRelaxVar.not())
  → 即: 当 relax_rule1 == false 时，约束正常执行
       当 relax_rule1 == true 时，约束被跳过（松弛）
```

**AlgCPLinearExpr**：线性表达式构建器，维护 `terms` 列表用于日志和展示。

`PartAlgCPLinearExpr` 扩展了此类型，额外维护 `PartTerm` 列表（携带组件分类/实例/代码元数据），用于优先级表达式的可读展示。

**变量初始化**（`ModuleBaseAlgImpl.initAll`）：

每个 `Para` 创建：

- `value` IntVar（枚举值或整数值）
- `opt_*` BoolVar 数组（枚举选项）

每个 `Part` 创建：

- `qty` IntVar（`[0, maxQuantity]`）
- `selected` BoolVar
- 联动约束：`selected == 1 → qty >= 1`；`selected == 0 → qty == 0`

### 6.4 执行器层（Executor）

`ModuleConstraintExecutorImpl` 是求解入口，其 `inferParas()` 方法编排完整求解流程：

```
1. 获取 Module（按 id 或 code）
2. 动态加载约束算法类（ModuleAlgClassLoader.newConstraintAlg）
3. 构建 ModuleInput
   ├─ cloneModule（克隆模块，隔离状态）
   └─ filterClone（按条件过滤 PartCategory）
4. 初始化 CP 模型
   ├─ new AlgCPModel()
   ├─ alg.init(model, module, moduleInput)
   │   ├─ initAll（创建所有变量）
   │   ├─ afterInitData（PartCategory 初始化分发）
   │   ├─ initInput（输入绑定）
   │   ├─ preCalculate（CalcStage.PRE）
   │   ├─ applyDecisionStrategies
   │   └─ initRules(CalcStage.MID)（执行业务规则）
   └─ alg.addRelaxObjectFunction()（添加松弛目标函数）
5. CpSolver 求解
6. 解写回（SouthboundInstViews）
7. POST 阶段后处理（ModulePostCalculator）
8. 返回 Result<SolverResult>
```

### 6.5 冲突诊断完整流程

当 `relaxSolve=true` 且原始模型无解时，引擎执行冲突诊断流程：

```
原始模型求解
    │
    ├─ OPTIMAL/FEASIBLE → 返回 SUCCESS
    │
    └─ INFEASIBLE
          │
          ├─ relaxSolve == false → 返回 NO_SOLUTION（空）
          │
          └─ relaxSolve == true
                │
                ├─ 第1次松弛求解
                │   ├─ 为所有可诊断约束附加松弛变量
                │   ├─ 目标函数: minimize Σ(relaxVar * weight)
                │   └─ 收集 relaxVar == true 的约束（第1批冲突）
                │
                ├─ 第2次松弛求解（带第1批冲突叠加权重）
                │   └─ 收集新的冲突约束（第2批）
                │
                ├─ 构建 DiagnosticConstraint 列表
                │   ├─ code: 规则编码
                │   ├─ constraintType: RULE / INPUT / SYSTEM
                │   ├─ naturalCode: 自然语言描述
                │   ├─ weight: 诊断权重
                │   └─ description: 可读说明
                │
                ├─ 提取部分可行解
                │   └─ 松弛模型有解时，solver → ModuleInst
                │
                └─ 返回 NO_SOLUTION
                      ├─ strictFeasible = false
                      ├─ diagnosticConstraints = [...]
                      └─ solutions = [部分解...]
```

**松弛目标函数的最小化**确保了求解器总是选择"违反最少、最轻微"的规则组合，这对应于**最小冲突集（Minimal Conflict Set）**的概念。

| **LP 松弛**        | LP Relaxation                         | 将整数变量替换为连续变量后的线性规划问题                     |
| **切平面**          | Cutting Planes                        | 用来切割 LP 松弛不可行区域的线性不等式                    |
| **Presolve**     | Presolve                              | 求解前的预处理阶段，简化问题规模                         |
| **变量域**          | Domain                                | 一个变量所有可能的取值集合                            |
| **组件分类**         | PartCategory                          | 一组相关组件的分类，支持多实例                          |
| **规则 Schema**    | RuleSchema                            | 规则的结构化表达类型                               |
| **本体**           | Ontology                              | 共享的概念模型，此处指 Onto 类                       |
