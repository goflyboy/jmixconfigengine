# 交互式 RFC 生成工作流

## 核心流程（4步迭代）

```
┌─────────────────────────────────────────────────────────────┐
│  步骤 1: AI写初稿                                           │
│    - 读取 TEMPLATE.md 获取标准格式                           │
│    - 读取 CORE-DESIGN.md 和 ACCEPTANCE.md 了解项目规范       │
│    - 根据用户需求生成RFC初稿                                 │
│    - 保存到 doc/RFC-XXXX-<title>.md                         │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  步骤 2: AI提出疑问                                         │
│    - 列出设计中的模糊点、技术决策点                          │
│    - 用 Q1, Q2, Q3... 编号                                  │
│    - 等待用户明确答复后再继续                                 │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  步骤 3: 用户检测并反馈                                     │
│    - 用户阅读初稿，提出问题或修改意见                        │
│    - AI根据反馈修改文档                                      │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  步骤 4: 重复迭代                                          │
│    - 重复步骤2-3，直到用户确认满意                           │
│    - 标记状态为「已评审」或「已批准」                        │
└─────────────────────────────────────────────────────────────┘
```

## AI评审机制

当用户要求「评审」或「审查」RFC时，执行以下流程：

1. **读取 REVIEW_PROMPT.md** 获取评审检查清单
2. **使用 Task 工具调用子代理**（generalPurpose模式）执行评审
3. **子代理读取 RFC 文档**，按照评审清单逐项检查
4. **输出评审报告**，包含：
   - 完整性检查
   - 一致性检查
   - 可实现性评估
   - 具体改进建议

### 调用外部AI评审示例

```

AI响应:
   请确认你希望使用哪个AI进行评审？
   - [ ] DeepSeek (当前对话)
   - [ ] Claude (创建子代理)

   选择后，我将启动评审流程...
```

## RFC 文档命名规范

```
doc/RFC-XXXX-<short-description>.md

示例:
- doc/RFC-0001-Decision-Strategy.md
- doc/RFC-0002-PartCategory-Filter-Empty-Handling.md
```

## 必需章节清单

每个RFC必须包含以下章节（详见 TEMPLATE.md）：

| 章节 | 说明 |
|------|------|
| 摘要 | 简明描述问题和解决方案 |
| 动机 | 问题背景、具体场景、为什么需要改变 |
| 设计方案 | 详细的技术方案，包含代码示例 |
| 验收准则 | 具体可测试的用例（必须包含测试代码） |
| 实现计划 | 分阶段任务列表和优先级 |
| 参考资料 | 引用的文档和外部资源 |

## 进度追踪

在RFC文档顶部使用状态标记：

```markdown
> 状态：草案（Draft）| 评审中（Review）| 已批准（Approved）| 已实现（Implemented）
> 日期：YYYY-MM-DD
```

## 关键约束

- **日志语言**：所有日志必须使用英文
- **不修改原文档**：评审意见输出到单独的文件或对话中
- **向后兼容**：确保设计不破坏现有功能
- **测试优先**：每个RFC必须包含可执行的测试用例
