# 交互式 RFC 生成工作流

                          │

│  步骤 1: AI写初稿                                           │

## AI评审机制

当用户要求「评审」或「审查」RFC时，执行以下流程：

1. **读取 REVIEW_PROMPT.md** 获取评审检查清单
2. **使用 Task 工具调用子代理**（generalPurpose模式）执行评审
3. **子代理读取 RFC 文档**，按照评审清单逐项检查
4. **输出评审报告**，包含：
   - 完整性检查
   - 一致性检查
   - 可实现性评估
   - 具体改进建议

## RFC 文档命名规范

```
doc/RFC-XXXX-<short-description>.md

示例:
- doc/RFC-0001-Decision-Strategy.md
- doc/RFC-0002-PartCategory-Filter-Empty-Handling.md
```

## 必需章节清单

每个RFC必须包含以下章节（详见 TEMPLATE.md）：

| 章节 | 说明 |
|------|------|
| 摘要 | 简明描述问题和解决方案 |
| 动机 | 问题背景、具体场景、为什么需要改变 |
| 设计方案 | 详细的技术方案，包含代码示例 |
| 验收准则 | 具体可测试的用例（必须包含测试代码） |
| 实现计划 | 分阶段任务列表和优先级 |
| 参考资料 | 引用的文档和外部资源 |

## 代码库侦察清单

生成 RFC 初稿前，必须先做一次轻量侦察。目标是少发明接口，多发现接口。

- **相似 RFC**：用 `rg "关键领域词" doc` 找已有设计。
- **相似测试**：用 `rg "inferRecommendModule|printSimpleSolutions|assertSoluContain" src/test/java` 找当前测试风格。
- **现有 helper**：优先找 `ModuleScenarioTestBase`、测试基类和已有 DSL，不先设计新的请求拼装 helper。
- **注解生成惯例**：确认字段名是否能推导 code，避免在 RFC 样例里重复写可推导字段。
- **实现入口**：查找相关 executor、builder、validator、alg model，确认 RFC 里的方法名和层次与当前代码一致。
- **用户偏好**：如果最近复盘或用户反馈已有明确偏好，将其写入 RFC 的设计约束。

侦察后在 RFC 中增加或更新以下小节：

```markdown
### 复用优先清单
- 优先复用：
- 不新增：
- 可由上下文推导：
- 相似测试来源：
```

## RFC 设计质量检查

生成或修改 RFC 时，除了写出主流程，还必须明确以下内容：

1. **分层边界**：说明规则或配置是在产品/模块基础数据阶段声明和生成，还是在销售/调用阶段使用；不要把规则声明、测试用例、运行时调用示例混成同一层设计。
2. **产品侧表达**：如果功能面向产品建模人员，RFC 中的注解、DSL、Schema 示例应使用业务友好的结构化表达，例如 `expr1 = "cpu.CoreNum=8"`，避免直接暴露内部实现字段。
3. **可推导信息**：列出哪些字段由解析器从子规则、表达式、`parentRuleCode` 或上下文推导。能推导的内容不应要求用户重复填写。
4. **接口矩阵**：列出所有受影响的入口，例如生成/推理、校验、序列化、注解生成、维护态校验。每个入口都要有对应验收用例或明确说明不在范围内。
5. **正反语义矩阵**：对每个枚举值或关系类型写清楚命中、不命中、边界条件的预期。例如组合规则必须写明 WHITE 和 BLACK 的默认语义。
6. **测试封装预期**：RFC 中的测试代码应优先展示测试基类 helper 和高层接口，不把大量请求对象拼装代码塞进具体测试方法。
7. **诊断语义**：若新增校验接口，RFC 应说明 boolean 结果、错误详情、violated rule code 等诊断信息如何返回和如何测试。
8. **复用优先清单**：列出应复用的现有类、方法、DSL 和测试基类；列出不应新增的 helper、DTO 或字段。
9. **可推导字段审查**：每个新增字段都要说明是否来自业务输入。能从字段名、category list、表达式或上下文推导的字段，不应要求用户重复填写。
10. **测试样例简洁性**：测试样例只表达本 RFC 关注点；不写未使用字段；part code 尽量短且有语义，便于断言直接使用 short code。

### 项目测试样例约束

在本项目中，RFC 的测试样例应优先贴近现有测试框架：

- 推荐/求解类测试优先使用 `inferRecommendModule(...)`。
- 单分类要求写成 `category:...`。
- 总体或跨分类要求写成 `category1,category2:...`。
- 不新增 `singleReqs`、`crossTotal` 等测试局部封装，除非现有基类无法表达。
- 测试样例默认调用 `printSimpleSolutions()` 方便调试。
- 断言优先使用语义化 short code；part code 足够短时直接复用字段名/code。
- 不写未使用的 `price` 等样板字段。

### 结构化规则 RFC 必填验收矩阵

涉及结构化规则、组合规则、注解规则或类似 Schema 展开的 RFC，验收准则至少覆盖：

| 场景 | 必测内容 |
|------|----------|
| 产品声明 | 通过注解/Schema/配置生成基础数据，而不是测试里手工拼内部规则 |
| 生成路径 | 原有推理或 InfoParse 风格入口仍能执行规则 |
| 校验路径 | validate/validator 入口返回 true/false，并能返回违规规则信息 |
| WHITE 语义 | 命中允许组合通过，未命中默认拒绝 |
| BLACK 语义 | 命中禁止组合拒绝，未命中默认通过 |
| 元数 | 二元和三元规则分别覆盖 |
| 边界 | 表达式无匹配、重复组合、运行时过滤、子规则不独立执行 |

## 进度追踪

在RFC文档顶部使用状态标记：

```markdown
> 状态：草案（Draft）| 评审中（Review）| 已批准（Approved）| 已实现（Implemented）
> 日期：YYYY-MM-DD
```

## 关键约束

- **日志语言**：所有日志必须使用英文
- **不修改原文档**：评审意见输出到单独的文件或对话中
- **向后兼容**：确保设计不破坏现有功能
- **测试优先**：每个RFC必须包含可执行的测试用例
