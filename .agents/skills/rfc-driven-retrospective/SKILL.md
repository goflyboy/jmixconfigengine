# RFC driven 复盘

## 目标

帮助团队在一次 RFC 驱动开发之后，系统地回答两个问题：

1. 怎样尽量一次性生成更可实现、更贴近项目现状的 RFC。
2. 怎样尽量一次性根据 RFC 生成更符合当前代码库、测试框架和用户偏好的代码。

复盘重点不是责备某次实现，而是发现缺失的知识，并把这些知识沉淀为 skill、项目文档、检查清单或协作约定。

## 使用原则

- 先列问题和改进方向；除非用户明确要求，不要直接改 skill 或文档。
- 以证据为基础：引用 RFC 内容、实际代码、测试写法、用户纠偏、提交 diff 或失败日志。
- 区分三类缺口：项目事实、用户偏好、流程护栏。
- 少做泛泛总结，多提可执行的改进项。
- 优先识别“为什么 AI 没有在动手前发现”，而不是只描述“后来改了什么”。

## 复盘输入

按需读取以下材料：

- RFC 文档：`doc/RFC-XXXX-*.md`
- 相关测试：尤其是用户指出的测试文件和相似测试，例如 `DynMultReq4MultiReqTest`
- 相关实现类：本次 RFC 修改过或用户指出的类
- 已有 skills：通常包括 `interactive-rfc-generation` 和 `rfc-driven-development`
- 用户反馈：本轮对话中用户指出的问题、偏好和纠偏点
- 代码状态：`git show`、`git diff`、测试输出、提交记录

不要一次性读取所有文档。先用 `rg` 找关键点，再读取相关片段。

## 复盘流程

### 1. 确定复盘范围

先明确本次复盘覆盖：

- RFC 生成阶段。
- RFC 到代码实现阶段。
- 测试设计和测试框架复用。
- 用户纠偏暴露出的知识缺口。
- skills 和项目知识结构的改进方向。

如果用户说“先列出来，不要改”，只输出清单，不改文件。

### 2. 建立“预期 vs 实际”对照

从 RFC 中提取：

- 核心设计决策。
- P0 范围。
- 验收准则。
- 测试样例。
- 建议新增的类、字段、helper。

再从代码和反馈中提取：

- 哪些实现后来被简化。
- 哪些字段、helper、封装被删除。
- 哪些测试写法被要求改为现有框架写法。
- 哪些断言、命名、调试习惯被用户补充。

输出时优先写“导致返工的差异”，而不是复述所有实现细节。

### 3. 分类缺失知识

把问题归入以下类别：

1. **项目事实缺失**：当前框架已有能力、隐藏规则、生成器惯例、DSL 能力、测试 helper。
2. **用户偏好缺失**：简洁性、少封装、少冗余字段、测试样例最小化、语义化 short code。
3. **流程护栏缺失**：实现前未做相似测试检索、未做字段可推导审查、未做最小新增代码挑战。
4. **RFC 表达缺失**：RFC 示例过度抽象、测试代码不贴近真实框架、P0 语义边界不清。
5. **skill 缺失**：现有 skill 说了原则，但没有变成可执行检查步骤。
6. **知识结构缺失**：缺少项目测试惯例索引、注解建模惯例、算法查找语义等长期知识。

### 4. 检查 RFC 生成阶段的一次性质量

重点检查 RFC 是否在生成前完成了足够侦察：

- 是否先读取 `CORE-DESIGN.md`、`ACCEPTANCE.md`。
- 是否查找相似 RFC 和相似测试。
- 是否确认现有 DSL 或 helper 能否表达需求。
- 是否列出“复用优先清单”和“不新增清单”。
- 是否明确哪些字段可由上下文推导。
- 是否使用项目真实测试写法，而不是临时 helper。
- 是否把用户侧表达和内部实现字段分清。

常见改进方向：

- 给 `interactive-rfc-generation` 增加“代码库侦察阶段”。
- 给 RFC 模板增加“复用优先清单”。
- 要求 RFC 测试样例引用相似测试来源。
- 加入样例简洁性检查：不用的字段不写，能推导的 code 不写。

### 5. 检查 RFC 到代码的一次性实现质量

重点检查实现前是否完成了“最小新增代码挑战”：

- 新字段是否必须存在，还是可以从已有字段推导。
- 新 DTO/helper 是否必须新增，还是现有 DSL 和基类已能表达。
- 查找逻辑是否符合当前运行时语义，例如精确 category code vs prefix/实例查找。
- 测试是否优先复用 `ModuleScenarioTestBase`。
- 断言是否使用语义化 short code。
- 推荐/求解类测试是否调用 `printSimpleSolutions()`。
- 回归测试是否覆盖相关旧功能。

常见改进方向：

- 给 `rfc-driven-development` 增加实现前知识卡片。
- 强制先 `rg inferRecommendModule` 和相似测试。
- 对每个新增字段写一句“为什么不能推导”。
- 对每个新增 helper 写一句“为什么现有 helper 不够”。

### 6. 提炼项目知识结构改进

优先考虑新增或更新这些知识：

- 项目测试惯例索引：
  - `inferRecommendModule` DSL 示例。
  - `assertSoluContain` 使用方式。
  - `printSimpleSolutions` 使用习惯。
  - short code 生成规则。
  - 多实例断言格式，例如 `I1_d2(...)`。
- 注解建模惯例索引：
  - 字段名如何推导 code。
  - `@PartAnno.code` 什么时候可省。
  - `fatherCode`、`required`、`supportMultiInst` 的使用原则。
  - 测试中未使用的 `price` 等字段不要写。
- 算法层查找语义索引：
  - 精确 category code 查找。
  - prefix/实例查找适用场景。
  - 逻辑 category 与多实例 category 的关系。
  - filtered candidate 与 original candidate 的边界。

## 推荐输出结构

使用中文输出，建议结构如下：

```markdown
## 复盘结论

一句话说明本次复盘发现的核心问题。

## 暴露出的缺失知识

1. 项目事实缺失：...
2. 用户偏好缺失：...
3. 流程护栏缺失：...

## 对 RFC 生成阶段的改进方向

- ...

## 对 RFC 到代码实现阶段的改进方向

- ...

## 对 skills 的改进方向

- interactive-rfc-generation: ...
- rfc-driven-development: ...

## 对知识结构的改进方向

- ...

## 建议下一步

只列建议，不直接改，除非用户要求。
```

## 项目特定检查点

在本项目中，RFC driven 复盘尤其要检查：

- 测试方法是否优先使用 `inferRecommendModule(...)`。
- 单分类要求是否写成 `category:...`。
- 总体要求是否写成 `category1,category2:...`。
- 是否避免新增 `singleReqs`、`crossTotal` 这类测试局部封装。
- 是否避免新增可由 category list、表达式或上下文推导的 code 字段。
- part 测试 code 是否短到能直接作为 short code。
- 断言是否使用业务可读的 short code，而不是无语义 `PTn`。
- 测试样例是否去掉未使用字段，如 `price`。
- 每个推荐/求解测试是否调用 `printSimpleSolutions()`。
- category code 查找是否是精确匹配，除非明确需要实例或 prefix 语义。

## 实现前知识卡片

如果复盘结论要转化为流程改进，建议让未来 RFC 实现前先填写：

```text
相似 RFC：
相似测试：
现有 helper：
现有 DSL：
现有注解/生成规则：
需要新增的字段：
可以推导的字段：
需要新增的生产类：
需要新增的测试 helper：
可能误用的旧逻辑：
回归测试范围：
用户偏好检查：
```

如果这张卡片填不出来，说明还不适合直接实现。
