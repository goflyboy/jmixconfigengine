# RFC 文档审查

## 总目标

用于审查并修订已经生成好的 RFC 或设计文档。审查时要同时看三类证据：正式 RFC 文档、SpecStory 沟通记录里的人工修正、当前代码库。默认审查后直接修改 RFC，并在修改前保留一份带“原文档”字样的备份。修改 RFC 时必须参考 `$interactive-rfc-generation` 的 RFC 质量要求。

## 硬性约束

- 默认审查后直接修改 RFC 文件。
- 修改前必须先备份原 RFC，备份文件名必须包含“原文档”。
- 如果用户明确说“不改”“先不要改”“只审查”“只提意见”“等我确认”，则只输出审查意见，不修改、不备份。
- 审查阶段不写实现代码。
- 用户在 SpecStory 里的人工输入优先级最高，尤其是用户对上一个大模型的纠偏、否定、补充和边界要求。
- 大模型输出只作为辅助背景；除一开始生成的 Doc/RFC 正文外，不要展开阅读大量模型回复。
- 当前代码证据优先于 RFC 中的想象性接口。
- 事实、推断、疑问要区分清楚；没有证据时标为“需确认”。
- 对证据充分、方向明确的问题直接修改；对需确认、存在歧义或可能改变产品决策的问题，不要强行改，放入“待用户确认”。
- 修改 RFC 时必须读取并参考 `.agents/skills/interactive-rfc-generation/SKILL.md`。如果该 Skill 中“评审不修改原文档”的约束与本 Skill 冲突，以本 Skill 的“备份原文档后默认修改”规则为准。

## 修改与备份规则

默认模式下，在编辑 RFC 前先复制原文档：

- 原文件：`doc/RFC-0011-RuleTrans-Module.md`
- 首选备份：`doc/RFC-0011-RuleTrans-Module-原文档.md`
- 如果首选备份已存在，使用时间戳避免覆盖，例如：`doc/RFC-0011-RuleTrans-Module-原文档-20260607-153000.md`

备份完成后再修改原 RFC 文件。不要修改备份文件。

## RFC 修订质量规则

修改 RFC 时，必须参考 `.agents/skills/interactive-rfc-generation/SKILL.md` 中的项目要求。不要机械复制全部内容，而是把相关要求落到当前 RFC：

- 章节完整性：尽量对齐 `摘要`、`动机`、`设计方案`、`验收准则`、`实现计划`、`参考资料` 等标准章节；缺失章节要补齐或说明不适用。
- 代码库侦察：修改前参考 `doc/CORE-DESIGN.md`、`doc/ACCEPTANCE.md`、相似 RFC、相似测试、现有 helper、现有 DSL 和实现入口。
- 复用优先清单：在 RFC 中明确 `优先复用`、`不新增`、`可由上下文推导`、`相似测试来源`，防止重复造接口、DTO、字段或测试 helper。
- 分层边界：区分产品/模型声明阶段、生成阶段、运行时求解、校验阶段和测试辅助逻辑，不把不同层次混在一个设计里。
- 产品侧表达：面向产品建模人员的注解、DSL、Schema 示例要使用业务友好表达，避免暴露不必要的内部实现字段。
- 可推导信息：能从字段名、分类列表、表达式、`parentRuleCode` 或上下文推导的信息，不要求用户重复填写。
- 接口矩阵：列出受影响入口，例如生成/推理、校验、序列化、注解生成、维护态校验；每个入口要有验收用例或明确不在范围内。
- 正反语义矩阵：对枚举值、关系类型、命中/不命中、边界条件写清预期。
- 测试封装预期：测试样例优先展示测试基类 helper 和高层接口，不把大量请求对象拼装写进具体测试方法。
- 诊断语义：如果新增校验接口，说明 boolean 结果、错误详情、violated rule code 等诊断信息如何返回和如何测试。
- 测试样例简洁性：测试样例只表达本 RFC 关注点，不写未使用字段；part code 尽量短且有语义。
- 项目测试习惯：推荐/求解类测试优先考虑 `inferRecommendModule(...)`、`printSimpleSolutions()`、语义化 short code，以及现有测试基类；不要轻易新增 `singleReqs`、`crossTotal` 等局部封装。
- 进度标记：如果 RFC 顶部缺少状态信息，按项目习惯补充或修正状态标记，例如 `草案（Draft）`、`评审中（Review）`、`已批准（Approved）`、`已实现（Implemented）`。

## 输入发现

### 1. 正式 RFC 文档

优先使用用户给出的文件路径，例如 `doc/RFC-0011-RuleTrans-Module.md`。如果用户没有给路径，用：

```bash
rg --files doc | rg "RFC-.*\.md$"
```

阅读 RFC 时重点看：

- 问题背景、目标、范围、非目标是否明确。
- 设计方案、接口、数据结构、示例代码是否互相一致。
- 验收标准是否可执行、可观察、覆盖正向、反向和边界场景。
- 实现计划是否能映射到当前代码位置。

### 2. SpecStory 沟通记录

沟通记录优先来自 `.specstory/history/*.md`。先用 RFC 编号、RFC 文件名、模块名、相关类名、用户提到的关键词搜索，例如：

```bash
rg "RFC-0011|RuleTrans|ModelHelper|ruletrans" .specstory/history
```

SpecStory 常见角色标记：

- `_**User`：人工输入，重点阅读。
- `_**Agent`：大模型输出，降权处理。
- `<tool-use ...>`：工具调用和日志，除非能证明代码或文档来源，否则通常跳过。
- `<think>` 或模型推理段：通常跳过。

处理规则：

- 先抽取所有人工输入片段，尤其是用户提出的目标、约束、纠偏、反对意见、补充边界。
- 对人工输入做小表格：原始诉求、用户修正点、设计含义、当前 RFC 是否落实。
- 只重点看大模型一开始给出的 Doc/RFC 正文或明确写入文件的文档内容。
- 不要把模型后续解释、辩解、工具日志当成用户需求来源。
- 如果模型输出和用户输入冲突，以用户输入为准。
- 如果 SpecStory 文件很长，先用 `rg "_\*\*User|_\*\*Agent|Doc|RFC|用户|修改|不对|不要|保持|新增"` 定位，不要整篇吞上下文。

### 3. 现有代码

根据 RFC 中出现的类名、方法名、包名、注解、DTO、测试 helper、模板和业务词搜索代码。优先用 `rg` 和 `rg --files`。

重点验证：

- RFC 提到的入口、类名、方法名、字段名、枚举值是否真实存在或合理。
- 是否已有 builder、parser、validator、executor、model、annotation、serializer 可以复用。
- 是否已有相似测试和测试基类；不要轻易接受 RFC 中新增测试 helper 的设计。
- 新功能应该放在哪个模块边界，是否会绕开已有架构。
- RFC 是否遗漏兼容性、迁移、诊断信息、失败行为和回归测试。

## 审查流程

### 第一步：正式文档审查

检查 RFC 自身是否完整、清晰、可落地：

- 动机是否对应真实问题。
- 方案是否解释了为什么这样设计。
- 新增接口和数据结构是否必要，是否有可从上下文推导却重复填写的字段。
- 示例代码是否与接口和当前工程风格一致。
- 验收标准是否覆盖主要路径、异常路径、边界路径。
- 是否区分产品/模型声明阶段、生成阶段、运行时求解、校验和测试辅助逻辑。

### 第二步：人工修正复核

从 SpecStory 的 `_**User` 片段中抽取用户修正点，并逐条判断：

- `已落实`：RFC 已正确吸收用户修正。
- `部分落实`：RFC 吸收了方向，但缺少细节、范围、测试或代码对齐。
- `未落实`：RFC 仍然遗漏或违背用户修正。
- `需确认`：用户原话存在歧义，或与代码证据冲突。

复核重点是“用户为什么要纠偏”，不是复述聊天。

### 第三步：代码对齐审查

用当前代码校验 RFC 是否可实现、是否符合本项目习惯：

- 找到相关实现入口、模型、执行器、生成器、模板、测试基类。
- 对比 RFC 中的方法名、字段名、包边界、生命周期假设。
- 找出重复造轮子、绕开已有 helper、把测试便利误写成生产设计的问题。
- 找出 RFC 漏掉的回归测试和风险路径。

### 第四步：备份并修订文档

默认模式下，审查完成后直接处理确定性问题：

1. 先按“修改与备份规则”创建原文档备份。
2. 修改原 RFC 文件，不修改备份。
3. 修改内容要能追溯到正式 RFC、SpecStory 人工输入或代码证据。
4. 不要把“需确认”的推断写进 RFC 正文；只在最终输出里列为待确认。
5. 修改后重新阅读相关章节，确认没有格式断裂、重复章节或自相矛盾。

如果用户明确要求不改，则跳过本步骤，只输出审查意见。

## 输出格式

默认用中文输出。

默认修改模式下，先说明备份和修改结果：

```text
结论：已完成审查并直接修订 RFC；原文档已备份到 <备份路径>；主要修改解决了哪些风险；仍有哪些问题需要用户确认。
```

然后按以下结构输出：

```markdown
## 已修改
- 问题标题
  - 依据：RFC 章节、SpecStory 人工输入片段、代码位置
  - 修改：说明已经改了 RFC 哪些内容

## 原文档备份
- <备份路径>

## SpecStory 人工修正复核
- 修正点：
  - 状态：已落实/部分落实/未落实/需确认
  - 处理：

## 代码对齐检查
- 已对齐：
- 已修正：
- 仍需补充验证：

## 待用户确认
- Q1 ...
```

如果用户明确要求不改，只输出审查报告，先给一句结论：

```text
结论：当前 RFC 是否可以进入修改/实现；主要风险是什么；是否需要先等用户确认。
```

然后按以下结构输出：

```markdown
## 必须修改
- [高/中/低] 问题标题
  - 依据：RFC 章节、SpecStory 人工输入片段、代码位置
  - 问题：说明设计漏洞、矛盾或遗漏
  - 建议：说明用户确认后应如何修改 RFC

## 建议修改
- ...

## SpecStory 人工修正复核
- 修正点：
  - 状态：已落实/部分落实/未落实/需确认
  - 复核意见：

## 代码对齐检查
- 已对齐：
- 不一致：
- 需要补充验证：

## 待用户确认
- Q1 ...
```

输出要具体、可操作。尽量引用文件路径、类名、方法名、RFC 小节名，不要泛泛而谈。

## 证据规则

- 本地代码证据尽量带文件路径和行号。
- SpecStory 证据要说明是“人工输入”还是“大模型输出”。
- 如果只是根据有限文件推断，要明确写“推断”。
- 不要因为措辞风格提出修改；重点审查设计正确性、实现可行性、测试完整性和用户修正是否被落实。

## 用户明确要求不改时

如果用户要求只审查、不修改，则审查报告输出后停止，等待用户确认。用户确认后：

1. 复述将应用哪些审查意见。
2. 按“修改与备份规则”先备份原文档。
3. 只修改已确认的问题。
4. 尽量保留 RFC 原结构和术语。
5. 修改后重新阅读相关章节，并说明具体改了什么。

如果用户否定某条意见，将该否定视为本轮后续工作的项目偏好。
